## Phase 1 — Verständnis

### 1.1 Zielzustand

Nach S2.1 existiert in `specs/sprint-2/page-inventory.md` eine vollständige,
git-diffbare Markdown-Tabelle mit allen Seiten, die die neue Praxis-Webseite
mindestens enthalten muss, bevor ein Go-Live auf `westend-hausarzt.com`
verantwortet werden kann. Jede Zeile ist eindeutig einem Template-Typ
(Vorgriff auf S2.2), einer Content-Strategie (E1-Hybrid+SEO) und einer
Priorität (Go-Live-kritisch vs. Post-Go-Live) zugeordnet.

Das Inventar ist die einzige belastbare Vorbedingung für:
- **S2.2** — Template-Typologie (aus Häufigkeiten im Inventar)
- **S2.3** — Implementierungs-Batches (Reihenfolge + Scope je Batch)
- **S2.4** — Hauptnavigation (Menü-Struktur aus Inventar ableiten)
- **S2.5** — QA-Audit (gegen Inventar-Prio-Liste)

### 1.2 Constraints

- **Keine Theme-Änderung** in S2.1 (weder CSS noch PHP noch `functions.php`).
- **Keine neuen WP-Pages** anlegen in S2.1.
- **Kein Content-Schreiben** in S2.1 (das gehört zu S2.3 je Batch).
- **Kein Crawl** von `westend-hausarzt.com` (E4c — nur URL + Typ erfassen,
  Content-Extrakt erst pro S2.3-Batch).
- **Keine YAML-Dateien** in `trunk/content/pages/` (E3a — reine Markdown-
  Tabelle, kein Trunk-Migrations-TODO; Cortex-Web-WP-Adapter für Praxis-Pages
  existiert noch nicht, darum hat CW-001 hier noch keinen Greifbereich).
- `tools/verify.sh` muss grün bleiben (wird nicht berührt, aber am Ende
  trotzdem laufen, um PXZ-E-001/PXZ-E-008-Regressionen auszuschließen).
- **Sprache:** nur DE. Mehrsprachigkeit ist Sprint 3.
- **DESIGN_GUIDELINES** unberührt (keine Design-Entscheidungen in S2.1).

### 1.3 Implizite Annahmen (explizit gemacht)

  1. `specs/sprint-2/README.md` Kandidaten-Liste (8 Kernseiten)

## Phase 2 — Lösungsdesign

### 2.1 Artefakt

**Eine einzige Datei:** `specs/sprint-2/page-inventory.md`

Git-diffbar, append-safe (neue Seiten kommen unten dran, bestehende Zeilen
werden nur in `status` / `anmerkungen` überschrieben).

### 2.2 Tabellen-Spalten (Pflicht, jede Zeile vollständig)

| Spalte | Pflicht | Werte |
|--------|:-:|-------|
| `#` | ✓ | laufende Nr. |
| `slug` | ✓ | finaler URL-Slug, kleinbuchstaben, `-`-getrennt, z. B. `ueber-uns` |
| `titel_de` | ✓ | Seitenname wie in Navigation angezeigt |
| `url_prod` | ✓ | aktuelle URL auf `.com` oder `—` wenn neu |
| `template_typ` | ✓ | einer aus §2.3 |
| `content_quelle` | ✓ | einer aus §2.4 |
| `prio` | ✓ | `P0` / `P1` / `P2` (§2.5) |
| `status` | ✓ | `todo` / `spec` / `impl` / `done` |
| `anmerkungen` | — | Besonderheiten, TBD-Flags, offene Fragen je Zeile |

### 2.3 Template-Typen (Vorgriff auf S2.2)

Acht Typen, abgeleitet aus Site-Struktur:

| Typ | Verwendung | Erwartete Häufigkeit |
|-----|------------|:-:|
| `standard` | generische Textseite (Praxis, Legal, 404) | 4–5 |
| `team` | Ärzte-Übersicht | 1 |
| `arzt` | Einzel-Arzt-Profil | 8 |
| `fachrichtung-landing` | Fachrichtungs-Übersicht | 1 |
| `fachrichtung` | Einzelne Fachrichtung | 8 |
| `kontakt` | Standorte + Formular | 1 |
| `sprechstunden` | Öffnungszeiten + Terminbuchung | 1 |
| `homepage` | bereits live (`template-homepage.php`) | 1 |
| `karriere` | bereits live (`template-karriere.php`) | 1 |
| `404` | Fallback | 1 |

### 2.4 Content-Quelle (E1-Hybrid+SEO)

Pro Seite genau einer der folgenden Werte — keine Mehrfach-Zuordnung,
keine leeren Felder:

- **`prod-seo`** — aktuelle `.com`-Texte als Basis übernehmen, **Stil
  entspannen** (Duz-Ansprache wo passend, kürzere Sätze, weniger Nominal-
  stil) und **SEO-optimieren**: Keyword-Struktur, `<h1>`-Eindeutigkeit,
  Meta-Title (≤60 Zeichen), Meta-Description (≤155 Zeichen), H-Hierarchie,
  interne Verlinkung. Dies ist die **Default-Strategie** für Bestandsseiten.
- **`neu`** — keine `.com`-Entsprechung vorhanden oder Prod-Version unbrauchbar
  (Beispiel: 404-Seite).
- **`legal-extern`** — Datenschutz + Impressum: Quelle noch offen (Anwalt /
  Generator wie e-recht24 / Prod-Übernahme). Entscheidung ist **S2.3 Batch A
  Spec**, nicht S2.1.

### 2.5 Priorität

- **`P0`** — MUSS vor Go-Live existieren und veröffentlicht sein.
  Umfasst: Homepage, Datenschutz, Impressum, Kontakt, Praxis, Team-Übersicht,
  Sprechstunden, 404, Fachrichtungen-Landing, Karriere.
- **`P1`** — SOLL vor Go-Live; Notfall-Fallback über Platzhalter möglich.
  Umfasst: Einzel-Arzt-Profile (Platzhalter-Foto zulässig), einzelne
  Fachrichtungs-Seiten (Kernfächer Innere + Allgemein vor Nischenfächern).
- **`P2`** — KANN nach Go-Live nachgezogen werden, ohne Kompromiss an
  Rechtssicherheit oder Auffindbarkeit.

### 2.6 Seitenliste (Zielzahl ≈ 26 Einträge)

Voraussichtliche Verteilung:

| Kategorie | Anzahl | Beispiele |
|-----------|:-:|----|
| Kernseiten | 8 | Startseite, Praxis, Team, Sprechstunden, Kontakt, Datenschutz, Impressum, 404 |
| Fachrichtungen | 9 | 1 Landing + 8 Einzelseiten |

### 2.7 Bewusste Ausschlüsse (explizit NICHT in S2.1)
