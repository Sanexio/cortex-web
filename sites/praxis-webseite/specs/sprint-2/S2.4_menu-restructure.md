## §1 Kontext & Motivation

### Dr.-Stracke-Direktive (Session-18-Ende, 2026-04-22)

> „Sobald Inhaltscluster fertig migriert sind, muss das Menü mitwachsen, bevor
> der nächste Cluster begonnen wird. Sonst sind Inhalte vorhanden, aber für den
> Patienten unauffindbar."

Aktuelles `wp_list_pages`-Fallback (Blocksy-Parent, `inc/components/menus.php:52`)
liefert die alphabetische Liste aller 177 Publish-Pages — seit S2.3-checkups
offensichtlich chaotisch und patientenunfreundlich.

### Ist-Befund (2026-04-22)

1. **`.pxz-nav-list` hart codiert** in `inc/homepage-data.php:131–137` mit
   5 Fragment-Hrefs (`#fachrichtungen`, `#team`, `#service`, `#standorte`,
   `#karriere`). Das sind **Homepage-Scroll-Anker**, keine Seiten-Links.
   Auf jeder Nicht-Home-Page (z. B. `/praxis/`) führen die Links ins Nichts.
2. **Kein Mobile-Menü.** `.pxz-nav-list { display: none }` bei `<1100px`;
   kein Burger, kein Drawer. Tablet + Phone haben nur Logo + CTA.
3. **`register_nav_menus(['primary','footer'])`** ist angemeldet (functions.php:94–97),
   aber **kein** WP-Menu zugewiesen. Deshalb greift im Blocksy-Footer der
   `wp_list_pages`-Fallback.
4. **Brand-Switch `/team/`** (Sanexio-Logo, DG §18.3) funktioniert und muss erhalten bleiben.

## §3 Architektur-Entscheidungen (F1–F7)

| # | Entscheidung | Kurz-Begründung |
|:-:|---|---|
| **F1b** | Nav-Daten als PHP-Array in `inc/nav-data.php` (Funktion `pxz_nav_data()`) | Git-tracked, deterministisch. Admin-pflegbare Lösung (F1a) erst, wenn WP-Menu-Plugin-Risiken bewertet sind. Trunk-YAML (F1c) in eigener Folge-Session. |
| **F2b** | Submenu-Hover auf Desktop (pure CSS + optional JS für Keyboard) | Standard, barrierefrei. Mega-Menu (F2c) überdimensioniert für 5 Kinder. |

### T3 — `template-parts/header-nav.php` umschreiben
- Daten-Quelle: `pxz_nav_data()[$lang]['primary']` statt `pxz_homepage_content()[$lang]['nav']`.
- Rendert `<ul class="pxz-nav-list">` mit `<li>`. Items mit `children` bekommen
  `<li class="has-children">` + `<button class="pxz-nav-sub-toggle">` + `<ul class="pxz-nav-sub">`.
- `.is-active` per URL-Vergleich `trim($_SERVER['REQUEST_URI'],'/') === trim($item['href'],'/')`.
- Falls aktives Item ein Submenu-Kind ist, bekommt auch das Parent `.is-active-parent`.
- Burger-Button (`<button class="pxz-nav-burger" aria-expanded="false" aria-controls="pxz-nav-drawer">`) rechts neben CTA.
- Drawer-Container `<div id="pxz-nav-drawer" class="pxz-nav-drawer" inert>` am Ende des `<header>`.
- Alle ARIA-Attribute: `role="navigation"`, `aria-label`, `aria-current="page"` auf aktivem Link.

### T4 — `assets/css/nav.css` erweitern
- Block **„Submenu (Desktop)"**: `.pxz-nav-sub` absolut positioniert, `opacity: 0; pointer-events: none` bis `li.has-children:hover` oder `li.has-children:focus-within`.
- Block **„Aktive Markierung"**: `.pxz-nav-list a.is-active` mit `color: var(--pxz-red)` + `border-bottom: 2px solid`. Analog `.is-active-parent`.
- Block **„Burger-Button"**: sichtbar `<1100px`, verborgen `≥1100px`. Icon via 3 `<span>` oder inline SVG.
- Block **„Drawer"**: `position: fixed; top: 0; right: 0; width: min(380px, 88vw); height: 100dvh; transform: translateX(100%); transition: transform 0.3s`. `[aria-expanded="true"]` setzt `transform: translateX(0)`.
- Block **„Drawer-Accordion"**: `.pxz-nav-drawer .has-children details summary` als Toggle; `details[open] ul` ausgefahren.
- Block **„Scroll-Lock"**: `body.pxz-nav-open { overflow: hidden }`.
- Block **„Reduced-Motion"**: `@media (prefers-reduced-motion: reduce) { .pxz-nav-drawer { transition: none } }`.

### T5 — JavaScript `assets/js/nav.js` (~60 Zeilen, vanilla)
- Event-Listener: Burger-Click → Toggle `aria-expanded`, Toggle `body.pxz-nav-open`, Toggle `inert` am Drawer.
- ESC-Keydown → close.
- Click auf Overlay-Backdrop → close.
- Submenu-Toggle (Desktop Keyboard): `button.pxz-nav-sub-toggle` → Toggle `aria-expanded` am Submenu-`<ul>`.
- Progressive Enhancement: Ohne JS funktioniert Drawer per `<details>`-Fallback auf Mobile (Drawer ist statisch offen unterhalb des Headers).

## §7 Risiken + Mitigationen

| # | Risiko | Wahrsch. | Impact | Mitigation |
|:-:|---|:-:|:-:|---|
| **R-1** | **Home-MD5-Null-Delta bricht**, weil Nav-HTML sich ändert (`#anchor` → `/seite/`) | **Sicher** | Hoch laut C9, aber in dieser Session gewollt | Explizit als **gewollter Delta** im Self-Check dokumentiert. Home-Hero-Region (nicht Nav) muss MD5-stabil bleiben → separater regionaler Screenshot-Vergleich (AK-11) |
| R-2 | Submenu-Hover bricht Keyboard-Access | Mittel | Mittel | `:focus-within` parallel zu `:hover`; JS-Fallback für Safari/älter |
| R-3 | Drawer-JS bricht auf iOS Safari (inert nicht supported < 17) | Niedrig | Niedrig | Polyfill `inert-polyfill` oder `aria-hidden="true"` + `tabindex="-1"` als Fallback |
| R-4 | WPML-URL-Mapping für EN/FR/ES nicht-trivial | Mittel | Niedrig | Für EN/FR/ES zunächst DE-Slugs als Fallback; WPML-Slug-Translation in Sprint 3 |
| R-5 | `/fachrichtungen/` + `/aerzte/` existieren nicht als Page → 404 | **Sicher** | Mittel | T9 entscheidet: (a) Stub-Page mit „Weitere Angebote folgen" anlegen, oder (b) Items provisorisch auf `/` mit Hinweis-Badge, oder (c) Items temporär entfernen bis Cluster migriert. Architekten-Tendenz: **(a) Stub-Page** (minimaler Aufwand, kein toter Link). |
| R-6 | Submenu-CSS beeinflusst andere Page-Templates (Spezifitäts-Kollision) | Niedrig | Niedrig | `nav.css` bleibt alleinige Quelle für `.pxz-nav*`; keine globalen Selektoren |
| R-7 | Scroll-Lock auf iOS verhindert Safari-Swipe-Back-Gesture | Niedrig | Niedrig | `touch-action: pan-y` auf body, Drawer-eigener Scroll-Container |
| R-8 | `position: fixed`-Drawer bricht Blocksy-Sticky-Header (z-index) | Mittel | Niedrig | Drawer `z-index: 100` > nav `z-index: 50`; Blocksy-Header bleibt ausgeblendet (`#header.ct-header { display: none }` aus S2.3-B) |
