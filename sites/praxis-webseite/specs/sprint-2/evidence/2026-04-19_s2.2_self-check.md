## In-Session-Bug + Fix (FK-3 Plausible Scheinlösung)

**Symptom:** Erste WP-CLI-Verifikation nach T6 zeigte 2 Phantom-Einträge in `wp_get_theme()->get_page_templates()`:
```
[404.php] => " header).
[functions.php] => "-Header automatisch.
```

**Ursache:** WordPress' Page-Template-Auto-Discovery scannt PHP-Files via Regex auf den String `Template Name:` (mit Doppelpunkt). Mein Header-Comment in `404.php` (`* NOT a Page Template (no "Template Name:" header).`) und mein Filter-Comment in `functions.php` (`// Karriere registriert sich via "Template Name:"-Header automatisch.`) enthielten den Trigger-String — WP las die Reststrings als Display-Namen.

**Fix:** Beide Comments umformuliert auf `Template-Name` (mit Bindestrich, ohne Doppelpunkt). Re-Verifikation grün — Phantom-Einträge weg.

**Hotfix-Commit:** `dd3e4e1` (Theme-Repo).

**Lessons-Learned-Kandidat (für FEHLERPROTOKOLL bei „Session beenden"):**

> **PXZ-E-009 (Kandidat) — Comment-Strings triggern WP Page-Template-Auto-Discovery**
>
> WP scannt jede PHP-Datei im Theme-Root auf den Regex `Template Name:` und nimmt den Reststring als Display-Namen. Das passiert auch in normalen Code-Comments. Lösung: Strings, die die Trigger-Phrase als Beispiel/Erklärung nutzen, müssen die Trigger-Form (mit Doppelpunkt) vermeiden — `Template-Name` mit Bindestrich ist sicher.
>
> **Klasse:** FK-3 Plausible Scheinlösung — Code wirkt richtig (sauberer Comment), produziert aber unsichtbaren WP-Bug.
> **Prüfpunkt (Pre-Flight für künftige Sessions):** Nach jeder Theme-Änderung `wp eval 'print_r( wp_get_theme()->get_page_templates() );'` prüfen — Output muss exakt die intendierten Templates enthalten, keine PHP-Files-aus-Versehen.

Aufnahme ins `_rules/FEHLERPROTOKOLL.md` erfolgt im „Session beenden"-Workflow Schritt 3 (Pattern-Optimierung), wenn Dr. Stracke zustimmt.

## Lessons Learned (S2.2-LL-1…5 — Kandidaten für Pattern/FEHLERPROTOKOLL)

- **S2.2-LL-1:** WP-CLI-Verifikation (`wp eval get_page_templates`) ersetzt Browser-Verifikation für die Template-Registrierungs-AK — schneller, deterministisch, in CI-fähig.
- **S2.2-LL-2:** Code-Comment-Strings können WP-Auto-Discovery-Regexes triggern (siehe PXZ-E-009-Kandidat). Generelle Vorsicht bei „Anleitungs-Comments", die Tag-Strings als Beispiel nennen.
- **S2.2-LL-3:** **Skelett-Disziplin**: Wenn Spec sagt „minimale Werte erlaubt", aber andere Files leer sind → konsequente Skelett-Disziplin gewinnt vor Punkt-Befüllung. Begründung: AK-9 bleibt sauber prüfbar, S2.3 wird die Stilisierung sowieso liefern.
- **S2.2-LL-4:** **Sprechende Slugs vor 3-Buchstaben-Kürzeln**: `home` (live) ist bereits sprechend — wir vereinheitlichen nicht auf das schlechtere Pattern (`kar`). S2.0b kann den Refactor mitziehen.
- **S2.2-LL-5:** **Spec-Aussagen im Self-Check verifizieren, nicht blind übernehmen** — die Spec hatte in AK-1-Erklärung off-by-one (Homepage in der „grep-Zähl"-Aussage vergessen). Self-Check fängt das ab.
