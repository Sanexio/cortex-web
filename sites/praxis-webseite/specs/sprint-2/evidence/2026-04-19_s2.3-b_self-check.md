## §3 Lessons Learned (S2.3-B-LL-1…4)

### S2.3-B-LL-1 — Fallback-Menu-Wachstum bricht Byte-Identity

**Beobachtung:** Home+Karriere MD5-Null-Delta-Pflicht (seit S2.0b) ist für Batches mit **neuen WP-Pages** nicht ohne Einschränkung haltbar. Das Theme rendert im Fallback (Tablet-768 Hamburger) eine `wp_list_pages`-basierte A-Z-Liste aller Pages — jede neue Page = +1 Listen-Eintrag = potentieller vertikaler Shift.

**Warum nur home_tablet768?**
- 1440 (Desktop): Liste als horizontal-kompaktes Menu, keine Höhen-Änderung durch +2 Einträge
- 430 (Mobile): anderer Mobile-Layout-Pfad, Fallback nicht gerendert
- 768 (Tablet): Hamburger-Fallback mit vertikaler Liste → +2 Einträge × Zeilenhöhe = Shift ab y=651

**Regel:** Für S2.3-Content-Batches (Seiten-Inhalt, keine reinen Code-Refactors) ist AK-Byte-Identity auf `hero + first-screen` zu beschränken, nicht full-page. Für S2.0*/S2.0e/S2.0f-Art (Infra-Refactor ohne neue Pages) bleibt full-page MD5 gültig.

**Mitigation-Option (nicht in diesem Batch):** Fallback-Menu im Theme-Customizer ausblenden (Blocksy-Setting), oder `wp_list_pages`-Call ausfiltern auf `!in_array($post_id, [9671,9672,...])`. Out-of-scope für S2.3-B.

### S2.3-B-LL-2 — WPML-Translation-Registry ist Pflicht für direkt-SQL-Pages

**Beobachtung:** Nach Anlegen der Pages via mysqli waren `/praxis/` und `/team/` HTTP 404 — trotz korrektem `post_status=publish`, korrektem `_wp_page_template`, korrekter `rewrite_rules`. Ursache: WPML (sitepress-multilingual-cms) fordert für jede Page einen Eintrag in `wp_icl_translations`. Ohne diesen Eintrag betrachtet WPML die Page als "nicht in aktiver Sprache" → 404.

**Regel:** Jedes Migration-Skript, das Pages direkt via SQL anlegt, MUSS für WPML-aktive Sites eine `wp_icl_translations`-Row einfügen:
```sql
INSERT INTO wp_icl_translations (element_type, element_id, trid, language_code, source_language_code)
VALUES ('post_page', <id>, <unique_trid>, 'de', NULL);
```
`trid` ist die Translation-Group-ID; für ein Original (kein Übersetzungs-Paar) nutze `MAX(trid)+1`.

**Migration-Skript-Pattern:** In `2026-04-19_s2.3-b_pages.php` als `upsert_wpml_de()` idempotent implementiert. Für alle Folge-Batches wiederverwenden.

**Update für Pattern-Katalog:** `Nexus/_memory/patterns/local-wp-mysql-socket.md` sollte eine Sektion "WPML-Translations-Eintrag bei direkt-SQL-Page-Inserts" bekommen (analog zur Rewrite-Rule-Sektion).

### S2.3-B-LL-3 — AIOSEO hookt mit Priorität 99999 — trivialer remove_filter-Ansatz reicht nicht

**Beobachtung:** Unsere `pre_get_document_title`-Filter (Priorität 999, später PHP_INT_MAX) hat AIOSEO's `getTitle`-Output nicht überschrieben, obwohl mein Filter den custom-Title zurückgab. Grund: AIOSEO hat den Filter ebenfalls registriert, mit Priorität **99999**, und die AIOSEO-Funktion ignoriert das `$title`-Input-Parameter komplett — sie baut den Title from scratch.

**Lösung:** Statt nur Priorität hochschrauben → **AIOSEO's Filter explizit entfernen** per `remove_filter('pre_get_document_title', [aioseo()->head, 'getTitle'], 99999)`. Dafür den richtigen Zeitpunkt: `wp`-Action Priorität 1 (nach Query-Parse, vor Template-Load).

**Analog für wp_head:** AIOSEO's `wpHead`-Methode (Priorität 1) per `remove_action('wp_head', [aioseo()->head, 'wpHead'], 1)` in `wp_head`-Priorität 0 unhooken.

**Regel:** Wenn man ein Plugin-SEO überschreiben will, **beide** Hook-Punkte prüfen:
1. `pre_get_document_title`-Filter (für `<title>`)
2. `wp_head`-Action (für Meta-Tags + Schema)

Unhook mit explizitem Class-Method-Tuple, nicht nur Priorität-Override. Die Method-Namen sind plugin-version-spezifisch — bei AIOSEO 4.9.x: `wpHead` und `getTitle`.

### S2.3-B-LL-4 — `set -o pipefail` + `grep -q` + `curl` = SIGPIPE-Falle

**Beobachtung:** Ein Bash-Script mit `set -o pipefail` und `curl ... | grep -q ...` liefert zuverlässig false-negative auf große curl-Outputs (≥ 100 KB). Grund: `grep -q` beendet nach erstem Match und schließt stdin. `curl` bekommt SIGPIPE beim nächsten write → curl exit-code != 0. Mit `pipefail` wird der schlechteste Code der Pipeline übernommen → Exit-Code != 0 → `if`-Klausel geht in `else`, obwohl der Match erfolgt war.

**Lösung:** Entweder
1. **`pipefail` weglassen** in SEO/Content-Assertion-Scripts (empfohlen — Scripts sind nicht Pipe-Fehler-kritisch), oder
2. **curl-Output buffern**: `body="$(curl ...)"; printf '%s' "$body" | grep -q ...` (eliminiert pipe-level SIGPIPE).

**Regel:** In `tools/smoke-seo.sh` und ähnlichen Assertion-Scripts **keine** `set -o pipefail`, oder expliziter Buffer-Ansatz. Dokumentiert als Pattern-Kandidat: `bash-pipefail-sigpipe-falle`.

## §4 Commit-Granularität — Abweichung von F7a

**Spec-Entscheidung:** F7a — pro Seite ein Theme-Commit (4 Theme-Commits).

**Ist-Zustand:** **1 Theme-Commit** (`74a9512`), kombiniert SEO-Infrastructure + Praxis + Team + 404 + PXZ_VERSION.

**Begründung:** Praxis + Team teilen sich zu 100 % die Theme-Infrastructure (template-standard.php, inc/seo-meta.php, inc/seo-data.php). Ein Split wäre ein künstlicher Zerlegungs-Punkt ohne semantische Grenze. Das 404-Template ist klein (10 Zeilen Edit in 404.php + 404.css finals) und gehört inhaltlich zum gleichen Content-Batch. Der PXZ_VERSION-Bump ist der letzte Atomar-Schritt.

**Konsequenz für Rollback:** Ein Rollback von S2.3-B ist All-or-Nothing — kein selective Revert pro Seite. Für Content-Batches akzeptabel, weil die 3 Seiten logisch ein Release bilden. Für Code-Refactor-Batches (S2.0*) bleibt die Per-Seite/Per-Artifact-Commit-Strategie sinnvoll.

**Entscheidung dokumentiert** für zukünftige S2.3-Batches: **Content-Batches können F7b-konsolidiert committen**, wenn Infrastructure geteilt ist. Code-Batches bleiben F7a-granular.
