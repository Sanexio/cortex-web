### 1.3 Implizite Annahmen (explizit gemacht)

- **A1 — Probe-Page wird `publish`, nicht `draft` — mit dokumentierter technischer Schuld.**
  Die S2.0b-F3-Entscheidung sah „dauerhaft draft" vor. S2.0e revidiert das bewusst auf `publish` mit folgender Begründung:
  1. **Local-WP ist nicht im Netz.** Die Site läuft nur auf `*.local`-Domain mit self-signed HTTPS. Kein Suchmaschinen-Index, keine Endnutzer.
  2. **Auth-Komplexität vs. Nutzen.** Preview-Nonce-Logik oder Puppeteer-Login erfordert entweder REST-Application-Password im Repo (Secrets-Risiko) oder Login-Flow im Tool (Wartungslast). Für eine **einzelne** Probe-Page ist das unverhältnismäßig.
  3. **Page bleibt praktisch unsichtbar:** kein Menü-Eintrag, kein Link von anderen Seiten, Slug deutet Probe-Zweck an (`s2-0b-probe`). Yoast/AIOSEO sind auf Local nicht aktiv → kein Sitemap-Auto-Include.
  
  **Technische Schuld + Upgrade-Trigger (architektonisch verbindlich):**
  Sobald **eine** der folgenden Bedingungen eintritt, MUSS die Probe-Architektur auf Auth-basierte Draft-Preview umgestellt werden:
  - ≥ 3 Probe-Pages existieren (skaliert dann, Investition lohnt sich)
  - Sprint 1 Staging-Setup wird reanimiert (Probe-Pages dürften NICHT in Staging publiziert sein)
  - DF-Produktions-DB kommt ins Spiel (jeglicher Publish-Status ist dort Tabu)
  
  Upgrade-Weg: Puppeteer-Login über Application-Password aus `.env.local` (REST-Auth-Bootstrap-Pattern existiert bereits in `Nexus/_memory/patterns/local-wp-rest-auth-bootstrap.md`), Page-Status zurück auf `draft`, Preview-URL via `?page_id=<id>&preview=true&preview_nonce=<x>`.
  
  Eintrag in `FEHLERPROTOKOLL.md` als `PXZ-T-001` (erste technische Schuld mit Trigger-Bedingung) wird in T8 angelegt.
- **A2 — `template-standard.php` rendert mindestens einen `.pxz-container`.**
  Bestätigt durch S2.2-Implementierung (Commit `6c02cb4`): Alle 8 neuen Skelett-Templates haben einen Container-Div + Hero-Section. Die Probe greift auf `.pxz-container`, `.pxz-hero` DOM-Elemente.
- **A3 — Klassen ohne DOM-Präsenz werden per Stylesheet-Rule-Match geprüft.**
  `.pxz-btn`, `.pxz-eyebrow`, Typografie-Klassen sind in einer leeren Page nicht zwingend im DOM. Diese Probe läuft über `document.styleSheets`-Introspection (siehe 2.3 Technik-Detail).
- **A4 — Bestehendes `probe-design.mjs` kann erweitert werden**, statt ein neues Script zu erzeugen. Es nutzt bereits Puppeteer + page-registry.mjs, und der page-registry-Mechanismus ist darauf ausgelegt, neue Pages aufzunehmen.
- **A5 — Split-Check bleibt in Bash.** Konsistent mit bestehendem §1 `grep_split`. Kein postcss, kein Node-Parser.

### 1.4 Fehlerklassen-Bezug

- **FK-2 Scheinverständnis:** Eine Probe, die nur `.pxz-container` testet, würde grün werden, auch wenn `.pxz-btn` oder `.pxz-eyebrow` gebrochen wäre. Gegenmaßnahme: **8 Klassen-Assertions über 4 Kategorien** (Container, Button, Eyebrow, Typografie, Hero, Section, Card).
- **FK-4 Iteration:** Split-Check nur auf 2 Dateien würde `components.css` ↔ `homepage.css`-Kollision (S2.0b-Hauptrisiko) nicht fangen. Gegenmaßnahme: paarweiser Check über alle 3 CSS-Dateien.
- **FK-5 Kontextverlust:** IST-Werte (F4a) statt Guideline-Werte (F4b) — die Probe pinnt exakt den aktuellen Stand, nicht ein theoretisches Ziel. Das vermeidet, dass eine Probe grün bleibt, obwohl `components.css` von der Guideline abweicht (oder umgekehrt: fälschlich rot wird, weil Guideline mehrdeutig ist).

### 2.2 §1b Split-Check-Erweiterung (Bash, in `verify.sh`)

**Neu nach `grep_split()`** (vor `reset_scope()`):

```bash
# === §1b Split-Location-Check zwischen CSS-Schichten (S2.0e) ===
grep_split_css() {
  echo ""
  echo "=== §1b Split-Location-Check CSS-Schichten (S2.0b-AK-10) ==="
  local components="$THEME_DIR/assets/css/components.css"
  local homepage="$THEME_DIR/assets/css/homepage.css"
  local karriere="$THEME_DIR/assets/css/karriere.css"
  local tmp_c tmp_h tmp_k
  tmp_c=$(mktemp); tmp_h=$(mktemp); tmp_k=$(mktemp)
  grep -oE '^\.pxz-[A-Za-z0-9_-]+' "$components" 2>/dev/null | sort -u > "$tmp_c"
  grep -oE '^\.pxz-[A-Za-z0-9_-]+' "$homepage"   2>/dev/null | sort -u > "$tmp_h"
  grep -oE '^\.pxz-[A-Za-z0-9_-]+' "$karriere"   2>/dev/null | sort -u > "$tmp_k"

  local pair_fail=0
  for pair in "components:homepage $tmp_c $tmp_h" \
              "components:karriere $tmp_c $tmp_k" \
              "homepage:karriere $tmp_h $tmp_k"; do
    set -- $pair
    local name=$1 a=$2 b=$3
    local dupes
    dupes=$(comm -12 "$a" "$b")
    if [ -n "$dupes" ]; then
      echo "  ✗ Doppelte Basis-Selektoren in [$name]:"
      echo "$dupes" | sed 's/^/    /'
      pair_fail=$((pair_fail + 1))
    else
      echo "  ✓ Keine doppelten Basis-Selektoren in [$name]"
    fi
  done
  rm -f "$tmp_c" "$tmp_h" "$tmp_k"
  [ "$pair_fail" -gt 0 ] && FAIL=$((FAIL + 1))
}
```

**Grep-Regex-Strategie:** `^\.pxz-[A-Za-z0-9_-]+` matcht nur **Basis-Selektoren am Zeilenanfang** (eine Selektor-Zeile pro Regel, wie in allen drei Dateien konsequent formatiert). Pseudo-Klassen (`:hover`, `:where()`), Descendants (`.pxz-home .pxz-eyebrow`) und Kombinatoren werden absichtlich nicht geschnitten — damit ist die Prüfung konservativ: sie meldet nur **echte Basis-Klassen-Dopplungen** (z. B. `.pxz-btn` in zwei Dateien), nicht legitime Page-Scope-Overrides wie `.pxz-home .pxz-eyebrow` vs. `.pxz-eyebrow`.

### 2.3 §3b Component-Probe (Puppeteer, via page-registry + probe-design.mjs)

**Ansatz:** Zero-Code-Change in `probe-design.mjs`. Neuen Page-Eintrag in `page-registry.mjs` ergänzen, fertig. `probe-design.mjs` iteriert schon generisch über alle Pages.

**Herausforderung:** Einige Assertions (`.pxz-btn`, `.pxz-eyebrow`, Typografie) treffen auf **leeren** Page-Content keine DOM-Elemente. Zwei Sub-Lösungen:

- **Sub-Option α (DOM-Approach):** Template-Content minimal in `template-standard.php` um einen versteckten Probe-Block erweitern (`<div class="pxz-probe-root" hidden>` mit je einem Eyebrow/Button/Typo-Element). Nachteil: Touch am Template = Theme-Commit = bricht C1.
- **Sub-Option β (Stylesheet-Rule-Match):** Probe nutzt `document.styleSheets`-Introspection statt `document.querySelector`. Greift auf die CSS-Regel, nicht auf gerenderte Elemente. Nachteil: probiert nicht das tatsächliche Rendering, misst nur die deklarierte Regel — genau das, was S2.0b wollte.
- **Sub-Option γ (Page-Content-HTML):** Probe-Page bekommt minimal HTML-Content (`post_content`) mit je einem Element pro Klasse, gepflegt per direktem mysql-UPDATE (nicht im Repo versioniert). DOM-Approach ohne Theme-Touch.

**Entscheidung:** **γ (Page-Content)** — DOM-Approach ist 1:1 kompatibel mit existierendem `probe-design.mjs`, kein Theme-Touch, und `post_content` ist der natürliche Ort für Probe-Markup. Der Content ist minimal:

```html
<div class="pxz-probe-root" style="position:absolute;left:-9999px;">
  <span class="pxz-eyebrow">probe</span>
  <h1 class="pxz-display">probe</h1>
  <h2 class="pxz-title-1">probe</h2>
  <h3 class="pxz-headline">probe</h3>
  <p class="pxz-body">probe</p>
  <a class="pxz-btn pxz-btn-primary">probe</a>
  <section class="pxz-hero">probe</section>
</div>
```

Off-Screen (`left:-9999px`), kein visueller Eindruck, aber im DOM greifbar für `getComputedStyle`. Ein T-Schritt macht den `post_content`-UPDATE per direktem mysql (analog S2.0b-T3-Pattern).

### 2.6 Probe-Page-Zustand — Migration-Skript (statt ad-hoc mysql)

**Architektur-Entscheidung (delegiert 2026-04-19):** DB-Eingriffe laufen **nicht** als einmaliger mysql-UPDATE, sondern als idempotentes Migration-Skript im Cortex-Web-Repo. Begründung:

1. **Reproduzierbar:** Bei DB-Reset oder Gerät-Wechsel reproduziert ein Skript-Aufruf den Probe-Page-Zustand. Ad-hoc UPDATE wäre verloren.
2. **Git-trackbar:** SQL-Änderungen sind ab jetzt versionierbar und reviewbar.
3. **Skalierbarkeit:** S2.3 wird pro Skelett-Template weitere Probe-Pages brauchen. Erstes Migrations-Skript legt das Pattern, Folge-Sprints erweitern die Migrations-Reihe.

**Neue Datei:** `tools/migrations/2026-04-19_s2.0e_probe-page-setup.sh`

**Inhalt (Vertrag):**
```bash
#!/usr/bin/env bash
# Migration: S2.0e Probe-Page-Setup
# Idempotent — setzt Page 9670 (Slug s2-0b-probe) auf publish + Probe-Markup.
# Reproducible für Dev-DB-Setup und Migrations-Historie.

set -euo pipefail

MYSQL="/Users/cluster-mini-02/Library/Application Support/Local/lightning-services/mysql-8.0.35+4/bin/darwin-arm64/bin/mysql"
SOCK_GLOB="$HOME/Library/Application Support/Local/run/*/mysql/mysqld.sock"
SOCK=$(ls $SOCK_GLOB 2>/dev/null | head -1)
PAGE_SLUG="s2-0b-probe"
PAGE_ID=9670

PROBE_HTML='<div class="pxz-probe-root" style="position:absolute;left:-9999px;"><span class="pxz-eyebrow">probe</span><h1 class="pxz-display">probe</h1><h2 class="pxz-title-1">probe</h2><h3 class="pxz-headline">probe</h3><p class="pxz-body">probe</p><a class="pxz-btn pxz-btn-primary">probe</a><section class="pxz-hero">probe</section></div>'

# 1. Page existiert?
EXISTS=$("$MYSQL" -S "$SOCK" -u root -proot local -N -e \
  "SELECT COUNT(*) FROM wp_posts WHERE ID=$PAGE_ID AND post_name='$PAGE_SLUG' AND post_type='page';")
[ "$EXISTS" = "1" ] || { echo "FAIL: Page $PAGE_ID ($PAGE_SLUG) not found"; exit 1; }

# 2. Idempotent UPDATE (tut nichts, wenn schon korrekt)
"$MYSQL" -S "$SOCK" -u root -proot local -e \
  "UPDATE wp_posts SET post_status='publish', post_content='$PROBE_HTML' \
   WHERE ID=$PAGE_ID AND (post_status!='publish' OR post_content!='$PROBE_HTML');"

# 3. Verify
STATUS=$("$MYSQL" -S "$SOCK" -u root -proot local -N -e \
  "SELECT post_status FROM wp_posts WHERE ID=$PAGE_ID;")
echo "OK: Page $PAGE_ID status=$STATUS, content-length=$(echo -n "$PROBE_HTML" | wc -c) bytes"
```

**Eigenschaften:**
- Auto-detect Socket-Pfad (robust gegen Local-Version-Updates)
- Existenz-Check vor UPDATE (Fail-fast bei Zustands-Drift)
- `WHERE ... OR content != ...`-Guard macht UPDATE idempotent (keine unnötigen DB-Writes)
- Keine Credentials im Repo — `root/root` sind Local-Defaults (dokumentiert in Pattern `local-wp-mysql-socket.md`), nur über Socket erreichbar, kein Netzwerk-Exposure
