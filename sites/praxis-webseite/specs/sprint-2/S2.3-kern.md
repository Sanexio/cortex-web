### §3.2 Google-Maps-Umsetzung

**Option A (Einfach):** `maps.google.com/maps/embed`-iframe mit `q=Grüneburgweg 12 Frankfurt` → zeigt nur Ziel. Entspricht aktueller Archive-contact-us-192-Lösung. **Vorteil:** 0 externe Dependencies. **Nachteil:** keine zusätzlichen Marker.

**Option B (Mit Markern):** Google Maps JavaScript API (mit API-Key) + manuelle Marker für U-Bahn + Tiefgarage + Parkhaus. **Vorteil:** alle gewünschten POIs sichtbar. **Nachteil:** API-Key nötig (Dr.-Stracke-Aufwand: Google-Cloud-Projekt + Key + Billing-Account), Laufzeit-Kosten bei > Free-Tier, DSGVO (Google-Server-Ping).

**Option C (Custom-Map via mymaps.google.com):** Dr. Stracke legt eine „My Map" an (GUI, 15 Min), 3 Marker setzt, Share-Link als Embed-iframe einsetzt. **Vorteil:** alle POIs, kein Code, kein API-Key, DSGVO-gleiche Situation wie A. **Nachteil:** Einmal-Aufwand bei Dr. Stracke (außerhalb meiner Reichweite), Pflege manuell.

**Architekten-Vorschlag:** **Option A in dieser Session** (schnell, alle Dependencies da), Option C als Folge-Nachtrag (kleiner Dr.-Stracke-Schritt, danach 1-Zeilen-Änderung am Template). In der Spec Option C als Todo-Marker dokumentieren.

## §6 Risiken + Mitigationen

| # | Risiko | Eintrittswahrscheinlichkeit | Auswirkung | Mitigation |
|---|---|---|---|---|
| R-1 | Home MD5-Delta durch Refactor (Array-Sortierung / Whitespace-Output abweichend) | Mittel | Hoch (Rollback nötig) | Pre/Post-Probe aller 3 Viewports; bei Delta: `var_export`-Vergleich alt vs. neu, Root-Cause fixen oder zurückrollen |
| R-2 | Karriere-Archive-MD5 vs. Live-DB-MD5 abweichend (S2.3-B hat nachträglich editiert) | Mittel | Niedrig (Diff zeigen, Dr. Stracke entscheidet) | Diff vor Aktion anzeigen, keine automatische Mutation |
| R-3 | WPForms-Marker `pxz_kontakt_form_v1` existiert nicht | Hoch | Mittel | Graceful Fallback: Text „Bitte telefonisch/per E-Mail kontaktieren" + Kontaktblock; Dr. Stracke kann später Form anlegen + Marker setzen |
| R-4 | Google-Maps iframe durch neue Browser-Richtlinien (Permissions-Policy) blockiert | Niedrig | Niedrig | Standard-Embed-URL, keine Third-Party-Cookies, `loading=lazy` |
| R-5 | Neue Page `sprechstunden` wird von Rewrite-Rules nicht erkannt (Pattern aus S2.0e bekannt) | Mittel | Mittel | Nach `wp_insert_post`: Rewrite-Rules flushen + curl-Test; bei 404 bekannter Fallback-Pattern aus `local-wp-mysql-socket.md` |
| R-6 | PXZ-E-009 Auto-Discovery: neue `inc/practice-data.php` triggert Phantom-Template | Niedrig | Niedrig | Keine `Template Name:` Comment-Strings in der Datei; Verify via `wp post-types` |
| R-7 | Sprachblöcke en/fr/es kopieren sich versehentlich als de | Niedrig | Mittel | Funktion return je Sprache, Tests über 4 Language-Codes |
| R-8 | Doctolib-Link veraltet / UTM-Parameter irrelevant | Niedrig | Niedrig | UTM aus Prod übernommen (extrahiert aus `westend-hausarzt.com`); Dr. Stracke kann bei Bedarf neu setzen |
| R-9 | Session zu groß für 1 Sitzung (4 Pages + Refactor) | Hoch | Niedrig | Atomare Commits (§3.3); Session-Ende nach jedem grünen Commit möglich |

## §8 Rollback-Plan

Falls ein Commit grün ist, der nächste aber verify.sh rotiert:

1. `git revert <hash>` im Theme-Repo.
2. PXZ_VERSION zurückrollen.
3. Bei DB-Mutationen: Snapshot vor Sprint 17 ist `_content-archive/` + optional `mysqldump` vor Session 17 (wird zu Beginn von Phase 3 erzeugt).
4. Für Page-Insert `sprechstunden`: `wp post delete <ID>` (wp-cli) oder mysqli `DELETE FROM wp_posts WHERE post_name='sprechstunden'`.
5. Dr. Stracke informieren + Root-Cause dokumentieren in FEHLERPROTOKOLL.md (PXZ-E-010+).
