- `de` für deutsche Pages (Cluster `kern`, `checkups`, `services`, `aerzte`, `fachrichtung`, `diagnostik`, sowie deutsche Pages im `legacy`-Cluster).
- `en`/`fr`/`es`/`unknown` nur innerhalb `i18n-dublette/`.

```markdown
  # ... weitere gefundene Keys, erfasst roh

# --- Bilder-Referenzen (keine Downloads) ---

# --- Integritätsbeweis ---

<p>Unsere Praxis liegt mitten im Frankfurter Westend...</p>
...
```

### §3.3 MD5-Contract

- `content_md5` im Front-Matter = MD5 der Bytes **nach** `<!-- BODY-BEGIN ... -->`-Marker-Zeile.
- Der Marker-Zeilen-Text ist **konstant** (byteweise identisch in jeder Datei), damit ein einfaches `tail -n +N` den Body extrahiert.
- Self-Check-Script liest pro Datei den Body, rechnet MD5, vergleicht mit Front-Matter-Eintrag, vergleicht zusätzlich mit DB-Query `SELECT MD5(post_content) FROM wp_posts WHERE ID = <id>`. Alle drei müssen gleich sein.

## §4 Generator-Skript

### §4.1 Ort + Benennung

`tools/migrations/2026-04-20_s2.3-d-p2_content-extract.php`

### §4.2 Ablauf

1. DB-Connect via mysqli + Local-WP-Socket (Pattern `local-wp-mysql-socket.md`).
2. Query aus `wp_posts` für alle Pages (publish + draft + private), LEFT JOIN `wp_postmeta` (Template), LEFT JOIN `wp_icl_translations` (`wpml_lang`). MySQL-8-`only_full_group_by`-Workaround (Pattern aus Phase 1).
3. Cluster- und Prioritäts-Zuordnung **1:1 aus Phase-1-Inventar-CSV lesen** (nicht neu berechnen — Konsistenz-Garantie). Pro ID: CSV-Zeile laden, Spalten übernehmen.
4. Pro Page:
   - Pfad berechnen: `_content-archive/<cluster>/<lang_or_status>/<slug>-<id>.md`.
   - Front-Matter zusammenbauen (YAML-Ausgabe über `yaml_emit` wenn verfügbar, sonst manuelle Serialisierung — deterministisch).
   - `content_md5` berechnen (`md5($post_content)`).
   - Wenn Datei bereits existiert: bestehenden `content_md5` aus Front-Matter lesen, vergleichen. Wenn gleich → skip. Wenn verschieden → overwrite (mit Log-Zeile).
   - Wenn neu → write.
5. Bilder-URLs via `preg_match_all('#<img[^>]+src=["\']([^"\']+)["\']#i', $content, $matches)` → eindeutige URLs → `image_urls`-Liste.
6. Meta-Whitelist: `_wp_page_template`, `_thumbnail_id`, `_yoast_wpseo_title`, `_yoast_wpseo_metadesc`, `_aioseo_title`, `_aioseo_description`, `_aioseo_og_*`, `_aioseo_twitter_*`. Zusätzliche `_aioseo_*`-Keys mit Wildcard erfasst.
7. `prod_url_guess`: Parent-Chain rekursiv traversieren, Slug-Pfad zusammenbauen, mit `https://westend-hausarzt.com/` präfixieren.
8. Am Ende: Inventar-CSV um Spalte `archive_path` erweitern (neuer Output: `page-inventory-full.csv.new`, dann atomic-rename über alte Datei).
9. Output-Counter: created / updated / skipped / errors. Am Ende: Summary-Zeile.

### §4.3 Invarianten

- Skript schreibt **nur** in `_content-archive/**` und `specs/sprint-2/S2.3-D/page-inventory-full.csv`.
- Skript liest nur lesend aus DB — **kein `INSERT`/`UPDATE`/`DELETE` auf `wp_posts`**.
- Bei Fehler auf einer Page: Log + continue, nicht abort — partielles Archiv ist besser als kein Archiv.
- Skript ist PHP-CLI-only (kein Browser-Aufruf). Shebang: `#!/usr/bin/env php`.

## §7 Risiken + Mitigationen

| # | Risiko | Wahrscheinlichkeit | Mitigation |
|:-:|---|:-:|---|
| R1 | Dateiname-Kollision bei gleichem Slug+ID (sollte nicht möglich sein — ID ist Primärschlüssel) | sehr niedrig | Skript checkt vor Write: existiert Datei mit gleichem Name aber anderer ID? Abort wenn ja. |
| R2 | Slug mit Umlauten/Sonderzeichen im Dateinamen | mittel | `sanitize_file_name()`-Äquivalent in PHP; Slug kommt aus `post_name` → bereits URL-sicher, aber Safety-Net. |
| R3 | YAML-Serialisierung des Bodys kollidiert mit `---`-Marker (Body enthält `---`) | niedrig | Body wird **nicht** in YAML embedded, sondern steht nach dem Front-Matter als roher Text. Marker-Zeile trennt eindeutig. |
| R4 | Body enthält Bytes, die Git als Binary interpretiert (z. B. eingebettete Encoding-Artefakte) | mittel | `.gitattributes`-Eintrag für `_content-archive/*.md` → `text=auto` + forcierte LF. |
| R5 | `yaml_emit` nicht verfügbar (PHP ohne ext-yaml) | mittel | Fallback: manuelle Serialisierung nach simpler YAML-Untermenge (strings quoted, Zahlen bare, Arrays als Inline-JSON-kompatibel). Deterministisch. |
| R6 | Parent-Chain-Rekursion bei zyklischer post_parent-Referenz (DB-Korruption) | sehr niedrig | Max-Depth 10, bei Erreichen: `prod_url_guess = null` + Warnung. |
| R7 | Meta-Whitelist übersieht projektspezifische SEO-Keys | mittel | Zusätzlich alle `_aioseo_*` + `_yoast_*` Keys via Wildcard erfasst. Neue Keys = Log-Hinweis. |
| R8 | Generator bricht mitten im Lauf ab (Timeout, DB-Verbindung) | niedrig | Pro Datei atomar schreiben (temp + rename); idempotenter Re-Run greift wieder. |
| R9 | Inventar-CSV-Erweiterung überschreibt existierende Spalte | niedrig | Vor Write: Header prüfen, ob `archive_path` schon Spalte ist → skip Header-Zeile, nur Daten-Spalte ergänzen. |
| R10 | 189 neue Dateien blähen Git-Repo auf | niedrig | Erwartung: ≈ 2–5 MB Gesamtgröße (189 × ~20 KB avg). Akzeptabel. Bei Bedarf später komprimierbar (`_content-archive.tar.gz`), aber jetzt: Klartext > Kompression. |

## §8 Rollback-Plan

### §8.1 Rollback-Szenario

Wenn nach Archive-Generierung etwas nicht stimmt (MD5-Mismatch, Parser-Fehler, Pfad-Fehler):

```bash
# Nichts ist in DB geändert — Rollback heißt nur Dateien entfernen.
cd ~/Cortex/projects/Cortex-Web
rm -rf sites/praxis-webseite/_content-archive/
git checkout HEAD -- sites/praxis-webseite/specs/sprint-2/S2.3-D/page-inventory-full.csv
# Skript bleibt (für Debug), Archive weg.
```

### §8.2 Entscheidungsbaum

| Situation | Aktion |
|---|---|
| AK-5 oder AK-6 fail (MD5-Mismatch) | Rollback, Fehler analysieren, Skript fixen, neu laufen |
| AK-3 < 189 | Fehler-Log durchsehen, betroffene Pages manuell prüfen, Skript fixen |
| AK-9 DELTA | Rollback + Ursachen-Analyse (sollte nicht passieren — kein DB-Content-Edit) |
| AK-12 12/12 | Kein Rollback nötig |

### §8.3 DB bleibt unberührt

Anders als Phase 1 gibt es hier **keine DB-Mutation** → kein mysqldump-Backup nötig. Das ist die architektonische Sicherheit dieses Batches.
