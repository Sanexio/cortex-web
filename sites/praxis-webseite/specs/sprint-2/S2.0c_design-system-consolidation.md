## Phase 1 — Verständnis

### 1.1 Zielzustand

Nach S2.0c existiert **eine verbindliche Wahrheits-Quelle** für alle
Design-Fragen der Praxis-Webseite, strukturiert nach einem **4-Schichten-Token-Modell**,
das ein späteres Ersetzen der visuellen Stack-Ebene (z. B. SF Pro → Inter Tight
zum Testen) ermöglicht, **ohne Komponenten oder Seiten anfassen zu müssen**.

Konkret:

- `DESIGN_GUIDELINES.md` v3.0 — **einzige** menschen-lesbare Autorität.
- `assets/css/tokens.css` v2 — maschinenlesbarer Spiegel.
- Homepage v2.7.5 + Karriere v2.6.0 bleiben **visuell byteidentisch** (MD5-Null-Delta).
- Neue Komponenten und Seiten (ab S2.1/S2.2/S2.3) lesen ausschließlich Tokens,
  nicht mehr Hex-Werte oder hartcodierte Paddings.

### 1.2 Constraints

- **Null visuelle Änderung** gegenüber v2.7.5. MD5-Shot-Vergleich auf 6 Shots
  (Home + Karriere × 3 Viewports) muss identische Hashes zu den v2.7.5-Baselines
  liefern — gleicher Beweis wie bei S2.0.
- **Kein Component-Refactor** in S2.0c (das ist S2.0b-Scope). Bestehende
  `.pxz-*`-Klassen bleiben unverändert.
- **Kein Typo-/Farb-/Spacing-Wechsel** — nur Umstrukturierung der Token-Ebene
  und Verankerung neuer Regeln.
- **Kein Breaking-Change am `PXZ_VERSION`-Bump-Schema** — v2.7.5 → v2.7.6
  (Patch-Bump, Architektur-Infra-Change, identisch zur S2.0-Logik).
- `tools/verify.sh` bleibt grün (§1 Split, §2 Reset, §3 Computed-Style 54/54,
  §4 Alignment 10/10).
- **Kein Eingriff in Sprint-1-Infrastruktur** (DF-Credentials, `.env.sprint1.local`).

### 1.3 Implizite Annahmen (explizit gemacht)

- `tokens.css` v1 aus S2.0 enthält aktuell 12 Primitives + 6 semantische
  Tokens in einer Schicht. Die Aufteilung in Schicht 1 (Primitives) und
  Schicht 2 (Semantic) ist **konzeptionell bereits vorhanden**, aber nicht
  sauber benannt — S2.0c macht die Trennung explizit.
- **SF Pro bleibt Font.** Apple HIG (v2.3 §2.1) ist Pflicht. Inter Tight
  wird nicht adoptiert — Cortex-DS ist Backstop, nicht Stack-Vorlage.
- **Rot + Amber bleiben die Akzent-Farben** der Praxis-Site (v2.3 §5.3,
  harte Mix-Trennung hell/dunkel).
- Die Cortex-DS-Prinzipien `--c-*`-Aliase werden **nicht** 1:1 kopiert.
  Stattdessen: eigener semantischer Namensraum `--pxz-c-*`/`--pxz-space-*`,
  der die Cortex-DS-Logik spiegelt, ohne mit Juvantis zu kollidieren.
- Juvantis-Webseite bleibt unberührt — Cortex-DS gilt dort als eigenes
  Design-System (Shopify-Theme-Klon, anderer Stack).

### 1.4 Fehlerklassen-Bezug (`_rules/WORKING_MODE.md`)

- **FK-4 Iteration:** verhindern, dass Padding-/Abstand-Konflikte erneut
  auftreten (PXZ-E-006 `.pxz-sect` 96 vs. 144 px). Durch Schicht-2-Semantik
  `--pxz-space-section-y` wird jede Sektion zwangsweise gleich gesetzt.
- **FK-5 Kontextverlust:** `DESIGN_GUIDELINES.md` v3.0 ersetzt v2.3 komplett;
  obsolete Tabellen bleiben nur mit ⚠️-Marker und PXZ-E-ID stehen (§16.6-Regel
  wird konsequent angewandt).
- **PXZ-E-001 (Split-CSS):** wird durch die Regel „Werte stehen nur in
  `tokens.css`, nicht in `homepage.css` oder `karriere.css`" architektonisch
  ausgeschlossen (Split = Verletzung einer neuen harten Regel).

## Phase 2 — Lösungsdesign

### 2.1 Das 4-Schichten-Token-Modell

```
┌─────────────────────────────────────────────────────────────┐
│  SCHICHT 1 — PRIMITIVES (harte Werte, stackspezifisch)      │
│  tokens.css :root                                           │
│  --color-red-600, --color-ink-900, --space-1 ... --space-14 │
│  --font-display-family, --font-display-weight-semibold      │
│  --radius-pill, --shadow-elevation-3, …                     │
│                                                             │
│  ⤷ Bei Stack-Austausch: NUR diese Schicht anfassen.        │
└─────────────────────────────────────────────────────────────┘
            │  semantischer Aufsatz (var-Referenz)
            ▼
┌─────────────────────────────────────────────────────────────┐
│  SCHICHT 2 — SEMANTIC (stackunabhängige Bedeutung)          │
│  tokens.css :root                                           │
│  --pxz-c-ink:       var(--color-ink-900);                   │
│  --pxz-c-accent:    var(--color-red-600);                   │
│  --pxz-c-surface:   var(--color-white);                     │
│  --pxz-space-section-y:   var(--space-13);    (≈96px)       │
│  --pxz-space-card-y-lg:   var(--space-14-72);               │
│  --pxz-font-display:      var(--font-display-family);       │
│                                                             │
│  ⤷ Komponenten lesen NUR diese Schicht.                    │
└─────────────────────────────────────────────────────────────┘
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│  SCHICHT 3 — COMPONENTS (immer semantisch adressieren)      │
│  assets/css/components.css (wird in S2.0b angelegt)         │
│  .pxz-card { padding: var(--pxz-space-card-y-lg)            │
│                        var(--pxz-space-card-x-lg); }        │
│  .pxz-btn--accent { background: var(--pxz-c-accent); }      │
│                                                             │
│  ⤷ Keine Hex-Werte, keine px-Zahlen mehr.                  │
└─────────────────────────────────────────────────────────────┘
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│  SCHICHT 4 — PAGES (komponieren)                            │
│  assets/css/homepage.css, karriere.css, …                   │
│  .pxz-hero .pxz-hero-sub { max-width: …; margin: auto; }    │
│                                                             │
│  ⤷ Ausnahmsweise Tokens, niemals Primitives.               │
└─────────────────────────────────────────────────────────────┘
```

**Austausch-Szenario Beispiel:**
„Wir wollen Inter Tight statt SF Pro." → Ein Edit in Schicht 1
(`--font-display-family: "Inter Tight", sans-serif;`), keine Änderung in
Schicht 2/3/4. `tools/verify.sh` Computed-Style-Probe bleibt grün (Padding-
und Farb-Werte bleiben), nur der Screenshot ändert sich — erwartet.

Ein Design-Wechsel (z. B. „wir probieren Inter Tight") läuft in 4 Schritten:
