## Phase 1 — Verständnis

### 1.1 Zielzustand

Nach S2.0f:

- Santapress ist aus der Local-WP-Installation **funktional entfernt**:
  - Nicht mehr in `wp_options.active_plugins` enthalten.
  - Plugin-Ordner `wp-content/plugins/santapress/` existiert nicht mehr im Live-Pfad.
  - WP-Rewrite-Rules neu generiert, enthalten keine Santapress-Handler mehr.
- Plugin-Code ist **archiviert, nicht gelöscht**: verschoben nach
  `sites/praxis-webseite/_archive/santapress-2026-04-19/` (reversibel per `mv` zurück).
- `tools/verify.sh` (§1 + §1b + §2 + §3 + §3b + §4) bleibt **Exit 0** (keine Regression).
- HTTP-Smoke-Skript prüft alle 10 Pages aus `page-registry.mjs` auf HTTP 200 (keine 500er durch fehlende Santapress-Hooks).
- Evidenz-Ordner `specs/sprint-2/evidence/2026-04-19_s2.0f_*` enthält:
  - Pre-Condition-Dump (`active_plugins` vor Eingriff, rewrite_rules-Count)
  - Post-Condition-Dump (gleiche Größen nachher)
  - `verify.sh`-Log vor und nach Eingriff
  - HTTP-Smoke-Log (10 URLs × HTTP-Status)
- Self-Check `evidence/2026-04-19_s2.0f_self-check.md` mit 6 AKs grün.

### 1.2 Constraints

- **C1 — Nur Local-WP.** `westend-hausarzt.com`-Produktion bleibt unberührt. Jeder Prod-Eingriff ist separate Entscheidung (SESSION_RESUME §4 P0d).
- **C2 — Kein Theme-Commit, kein PXZ_VERSION-Bump.** Theme-HEAD bleibt `8f596f7`, Version `2.7.8`.
- **C3 — Rollback-Fähigkeit garantiert.** Archive muss vollständig und per einzelnem `mv`-Befehl reversibel sein. Keine Teil-Löschung, keine selektive Datei-Entfernung.
- **C4 — Verify bleibt grün.** `verify.sh` vor + nach Eingriff Exit 0. Bei Regression: sofortiger Rollback.
- **C5 — Keine DB-Exports im Repo.** Pre/Post-Dumps werden als Text-Snippets in Evidenz-Markdown eingebettet (nur betroffene Felder), nie als voller DB-Dump.
- **C6 — Backup der `active_plugins`-Option vor Eingriff.** SQL-Statement + Vorher-Wert in Evidenz dokumentiert, damit Rollback ohne DB-Dump möglich ist.
- **C7 — Idempotent.** Wiederholtes Ausführen der Remove-Sequenz darf nicht fehlschlagen (z.B. wenn Plugin schon entfernt wurde).
- **C8 — HTTP-Smoke ist Regressions-Indikator, nicht Content-Validator.** Erwartet HTTP 200 (oder 301/302 bei registrierten Redirects) auf allen 10 Registry-Pages. Content-Korrektheit prüft `verify.sh`.

### 1.3 Implizite Annahmen (explizit gemacht)

- **A1 — Santapress ist regulärer Plugin-Ordner, nicht mu-plugin.**
  Bestätigt durch `ls wp-content/plugins/santapress/` (21 MB, mit `santapress.php` als Haupt-File). Kein Eintrag in `wp-content/mu-plugins/`.
- **A2 — Aktivierungseintrag hat Standardform `santapress/santapress.php`.**
  Wird in T1 (Pre-Condition-Snapshot) verifiziert. Falls abweichend, wird die Spec aktualisiert, bevor T3 läuft.
- **A3 — Theme-Repo enthält keine Santapress-Referenzen.**
  Verifiziert via `grep -ri "santapress" wp-content/themes/praxiszentrum/` → 0 Treffer. Entfernung erfordert keine Theme-Änderung.
- **A4 — Rewrite-Rules-Neuanlage erfolgt via DELETE + Warmup-Request.**
  Bewährter Workflow aus `local-wp-mysql-socket.md` §Grenze. WP-CLI ist an Local-by-Flywheel-Socket unzuverlässig (S2.0b-LL-5) → direkt-mysql + Warmup.
- **A5 — Keine andere aktive Komponente hängt an Santapress-Hooks.**
  Wird in T2 verifiziert (grep über alle anderen aktiven Plugins + WP-Core auf `santapress`-Hook-Namen).
- **A6 — `generateblocks`, `freesia-empire-plus`, `theme-freesia-demo-import` bleiben unberührt.**
  Sie sind potenzielle Phantom-Template-Emitter (SESSION_RESUME §4 P3), aber **außerhalb** des S2.0f-Scopes.

## Phase 2 — Lösungsdesign

### 2.1 Architektur der Entfernung (5 atomare Schritte)

```
T1  Pre-Condition-Snapshot    →  DB-Zustand + Dateisystem vor Eingriff festhalten
T2  Hook-Dependency-Audit     →  andere Plugins/Theme auf Santapress-Referenzen prüfen
T3  Deaktivierung + Archive   →  active_plugins anpassen, Ordner nach _archive/ mv
T4  Rewrite-Rules regenerieren →  DELETE + Warmup-curl auf Home
T5  Verify-Gate                →  verify.sh + HTTP-Smoke auf 10 Pages
```

Jeder Schritt ist rollback-fähig:
- T3 rückgängig: `mv _archive/santapress-2026-04-19/ wp-content/plugins/santapress/` + active_plugins-UPDATE mit Pre-Condition-Wert aus T1.
- T4 rückgängig: nicht nötig (rewrite_rules werden beim nächsten WP-Request ohnehin neu generiert).

### 2.2 Gewählte Strategie: Archive-Then-Activate-Off vs. Active-Off-Then-Archive

**Entscheidung: Active-Off ZUERST, dann Archive (S2.0f-DEC-1).**

Reihenfolge: `active_plugins`-Update **vor** `mv`. Grund: Wenn der Ordner entfernt wird, während das Plugin noch in `active_plugins` steht, wirft WP bei jedem Request einen PHP-Warning (`Plugin file does not exist`). Das würde den HTTP-Smoke in T5 verschmutzen und die Regressions-Erkennung erschweren.

Alternative (Archive zuerst, dann active_plugins-Update): verworfen wegen obiger Warnings und instabilem Zwischenzustand.

### 2.3 Technik-Details je Schritt

**T1 — Pre-Condition-Snapshot** (Bash + mysql)
```bash
SOCKET="/Users/cluster-mini-02/Library/Application Support/Local/run/<site-hash>/mysql/mysqld.sock"
DB="local"
mysql --socket="$SOCKET" -uroot -proot "$DB" -e \
  "SELECT option_value FROM wp_options WHERE option_name='active_plugins'\G" \
  > evidence/2026-04-19_s2.0f_pre_active_plugins.txt
mysql --socket="$SOCKET" -uroot -proot "$DB" -e \
  "SELECT LENGTH(option_value) AS rewrite_rules_size FROM wp_options WHERE option_name='rewrite_rules'" \
  > evidence/2026-04-19_s2.0f_pre_rewrite_rules_size.txt
ls -la wp-content/plugins/santapress/ | head -5 > evidence/2026-04-19_s2.0f_pre_plugin_dir.txt
```
Erwartung: `santapress/santapress.php` enthalten im `active_plugins`-Serialize-String.

**T2 — Hook-Dependency-Audit** (Grep)
```bash
# Im live-Plugins-Verzeichnis (ohne santapress selbst)
grep -rli "santapress" wp-content/plugins/ \
  --exclude-dir="santapress" > evidence/2026-04-19_s2.0f_dep_plugins.txt

# Theme (erwartet leer)
grep -rli "santapress" wp-content/themes/praxiszentrum/ \
  > evidence/2026-04-19_s2.0f_dep_theme.txt

# mu-plugins (falls vorhanden)
grep -rli "santapress" wp-content/mu-plugins/ 2>/dev/null \
  > evidence/2026-04-19_s2.0f_dep_muplugins.txt || true
```
Abbruchbedingung: Wenn **anderes** Plugin (nicht Santapress selbst) auf `santapress`-Hooks referenziert → Stopp, Spec anpassen, Dr. Stracke entscheiden lassen.

**T3 — Deaktivierung + Archive** (mysql + mv)
```bash
# 3a. Vorher-Wert sichern (schon in T1, hier als Backup-Befehl für Rollback)
mysql ... -e "SELECT option_value FROM wp_options WHERE option_name='active_plugins'" \
  > evidence/2026-04-19_s2.0f_active_plugins_backup.sql.txt

# 3b. Santapress aus Serialize-Array entfernen
#     Vorgehen: PHP-Unserialize → Array-Filter → PHP-Serialize → UPDATE
php -r '
$sock = $argv[1]; $db = $argv[2];
$mysqli = new mysqli(null, "root", "root", $db, 0, $sock);
$res = $mysqli->query("SELECT option_value FROM wp_options WHERE option_name=\"active_plugins\"");
$row = $res->fetch_assoc();
$arr = unserialize($row["option_value"]);
$new = array_values(array_filter($arr, fn($p) => stripos($p, "santapress/") !== 0));
$serialized = $mysqli->real_escape_string(serialize($new));
$mysqli->query("UPDATE wp_options SET option_value=\"$serialized\" WHERE option_name=\"active_plugins\"");
echo "removed: " . (count($arr) - count($new)) . " entry/entries\n";
' "$SOCKET" "$DB"

# 3c. Archive-Verschiebung
ARCHIVE_DIR="~/Cortex/projects/Cortex-Web/sites/praxis-webseite/_archive/santapress-2026-04-19"
mkdir -p "$(dirname "$ARCHIVE_DIR")"
mv wp-content/plugins/santapress/ "$ARCHIVE_DIR/"
```
Idempotenz: Wenn Santapress schon nicht mehr in `active_plugins` steht, gibt T3b `removed: 0` aus und fährt fort. Wenn der Plugin-Ordner schon weg ist, `mv` schlägt fehl — dann wird das als "schon erledigt" gewertet, nicht als Fehler.

**T4 — Rewrite-Rules regenerieren**
```bash
mysql ... -e "DELETE FROM wp_options WHERE option_name='rewrite_rules'"
# Warmup: 2 Requests auf Home, damit WP neue Rules schreibt
curl -sk -o /dev/null -w "%{http_code}\n" "$BASE_URL/"
curl -sk -o /dev/null -w "%{http_code}\n" "$BASE_URL/"
# Rules-Size danach prüfen
mysql ... -e "SELECT LENGTH(option_value) FROM wp_options WHERE option_name='rewrite_rules'" \
  > evidence/2026-04-19_s2.0f_post_rewrite_rules_size.txt
```
Erwartung: HTTP 200, Rules-Größe > 0 und ≠ Pre-Wert.

**T5 — Verify-Gate (F3b = verify.sh + HTTP-Smoke)**

Zwei Teile:

*(a) Bestehender `tools/verify.sh`* — unveränderter Lauf, erwartet Exit 0.

*(b) Neuer HTTP-Smoke — `tools/smoke-http.sh`* (neu anzulegen):
```bash
#!/usr/bin/env bash
# Smoke-HTTP: 10 Registry-Pages + Home, erwartet 2xx/3xx, keine 5xx.
set -euo pipefail
BASE_URL="${BASE_URL:-https://gpmedicalcenterwestend-7ded2f4ae8c4343d2029-202604.local}"
PAGES=(
  "/"
  "/karriere/"
  # 8 weitere Slugs aus page-inventory.md P0, zu bestätigen in T5-Vorbereitung
)
FAIL=0
for path in "${PAGES[@]}"; do
  code=$(curl -sk -o /dev/null -w "%{http_code}" "$BASE_URL$path")
  case "$code" in
    2*|3*) echo "  ✓ $code  $path" ;;
    *)     echo "  ✗ $code  $path" ; FAIL=1 ;;
  esac
done
exit $FAIL
```
Skript liegt unter `tools/smoke-http.sh`, ausführbar (`chmod +x`). Wird **nicht** in `verify.sh` integriert (Scope-Disziplin, S2.0e-LL-4); wird per separater Konsol-Anweisung aufgerufen.

### 2.4 Architekten-Entscheidungen F5 / F6 / F7 (delegiert durch Dr. Stracke, 2026-04-19)

**F5 = F5b, modifiziert (5xx-Fail-Regel):**
`smoke-http.sh` prüft alle 10 Registry-Pages, aber Fail-Kriterium ist ausschließlich **HTTP 5xx**. Codes 2xx/3xx/4xx sind OK. Grund: 404 auf den 8 Skelett-Template-Slugs ist erwartetes Verhalten (Pages noch nicht angelegt). 5xx wäre das einzige echte Regressions-Signal eines Santapress-Hook-Mangels. Nebeneffekt: Tool wird bei fortschreitender S2.3-Content-Arbeit automatisch aussagekräftiger, ohne Anpassung.

**F6 = F6b (Verfallsdatum):**
`_archive/santapress-2026-04-19/` enthält eine `ARCHIVE_README.md` mit Verfallsdatum **2026-05-19** (30 Tage). Wenn bis dahin keine Santapress-Abhängigkeit auffällt, wird das Archive in einer Folgesession gelöscht. Zwingt zu bewusster Entscheidung, verhindert Archive-Wildwuchs.

**F7 = F7b (erweiterte Evidenz):**
Zusätzlich zu `active_plugins` und `rewrite_rules`-Größe wird ein `wp_posts`-Count je `post_status` als Snapshot genommen (vor + nach). Beweist Unversehrtheit der Content-Objekte gegen potenzielle `deactivation_hook`-Nebeneffekte.

## Risiken + Mitigationen

| Risiko | Wahrscheinlichkeit | Impact | Mitigation |
|---|:-:|:-:|---|
| R1 — Anderes Plugin hängt an Santapress-Hook | niedrig | mittel | T2-Audit vor T3; bei Treffer Stopp |
