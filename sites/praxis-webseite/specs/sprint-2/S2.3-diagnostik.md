## §4 Risiken + Mitigationen

| # | Risiko | Mitigation |
|---:|---|---|
| R-1 | **Nested-URLs greifen nicht** — `post_parent` gesetzt, aber `rewrite_rules` nicht regeneriert, Kinder liefern 404 | Nach Migration `flush_rewrite_rules(true)` via WP-CLI ODER PHP-Call. Verify-AK-4 prüft Realzustand. Rollback: Parent auf 0 zurücksetzen, alte Slugs restaurieren |
| R-2 | **Slug-Kollisionen** — `labor` bereits vergeben, `sonographie` bereits vergeben | Pre-Check: `SELECT ID FROM wp_posts WHERE post_name='labor' AND post_type='page'`. Bei Fund abbrechen und per Dry-Run protokollieren |
| R-3 | **301-Redirects brechen** — WP nutzt `_wp_old_slug` für alte → neue Weiterleitung, funktioniert nur bei direktem Slug-Wechsel auf derselben ID. Bei Parent-Wechsel (Kind bekommt `post_parent`) wird der alte flache Slug nicht automatisch redirected | Manuell `_wp_old_slug` post_meta anlegen für die 3 Kinder (`schilddruesen-sonographie`, `ultraschall-der-beingefaesse`, `sono-atlas`). AK-2 prüft |
| R-4 | **Labor-Merge Idempotenz** — Script läuft zweimal, fusionierter Content wird doppelt eingefügt | Hash-Vergleich: erwarteter Post-Merge-MD5 gegen aktuellen MD5. Bei Match `skipped`. Pattern wie Impfungen-Merge S20 |
| R-5 | **Mojibake in `labordiagnostik-296` + `rund-ums-labor-475`** — UTF-8-doppelt-encoded ODER echte Replacement-Zeichen | `mb_convert_encoding($c, 'UTF-8', 'Windows-1252')` wo double-encoded, manuelles Char-Mapping bei echten `?`. Pattern `wp-mojibake-mb-convert-fix.md` (S15) |
| R-6 | **Menü wird bei Desktop unter ~1200 px zu breit** (8 Top-Items + 2 Submenus) | Verify-Screenshot bei 1440/1200/1024/768. Bei Bedarf: Items enger setzen via `gap`-Reduktion (nur Token-Anpassung, keine Inline-CSS) |
| R-7 | **🔴 DSGVO/Schweigepflicht bei sono-atlas-Befundbildern** — § 22 KunstUrhG (Recht am eigenen Bild), DSGVO Art. 9 (Gesundheitsdaten), § 203 StGB (Ärztliche Schweigepflicht). Selbst anonymisierte Ultraschall-Befunde können personenbezogen sein, wenn Aufnahmekontext identifizierend ist (Datum, Praxis, Organ-Ausschnitt + Patientengeschichte) | **Vor Go-Live:** Dr. Stracke bestätigt schriftlich (im Chat reicht), dass (a) die 5 Befund-Fotos patienten-anonymisiert sind (keine Namen, keine ID in Meta-Daten, keine datumsgebundenen Kontexte) UND (b) Einwilligungen der dargestellten Patienten dokumentiert vorliegen ODER die Aufnahmen so abstrakt sind, dass Re-Identifikation praktisch ausgeschlossen ist. **Falls unklar:** Migration der 4 Content-Pages ohne sono-atlas durchführen, atlas bleibt als Draft bis Klärung. Fallback: Stock-Befundbilder aus lizenzierter Medizin-Datenbank |
| R-8 | **Nav-Array-Erweiterung auf 8 Items** — Dr.-Stracke-Kommentar „Design anpassen beim Feintuning" berührt das bereits | Keine proaktive Design-Änderung in dieser Spec. S2.4-Polish bleibt separate Front. R-6 adressiert nur Layout-Break |
| R-9 | **WPML-Einträge-Konflikt** — neue Page `/diagnostik/` braucht trid, 9 UPDATE-Pages ebenfalls | Pre-Check: für jede ID `SELECT COUNT FROM wp_icl_translations WHERE element_id=$id`. Fehlende INSERT mit `MAX(trid)+n`. Pattern wie S2.3-aerzte-services |
| R-10 | **`/fachrichtungen/` + neue Diagnostik-Menü-Position** — D-2 setzt Diagnostik zwischen Check-Ups und Sprechstunden, aber `Fachrichtungen` steht davor | Akzeptiert (bewusste Wahl D-2). Menü-Screenshot-Review in AK-7 |

## §6 Rollback-Strategie

1. **DB-Rollback:** mysqldump-pre (T1) in Rollback-Datei. `mysql < rollback.sql` stellt exakten Zustand vor Migration wieder her.
2. **Theme-Rollback:** `git reset --hard <PRE-COMMIT-HASH>` auf Theme-HEAD `c090173` (PXZ 2.7.17) zurück.
3. **Cortex-Web-Rollback:** `git reset --hard <PRE-COMMIT-HASH>` auf Spec-Vor-Commit.
4. **Rewrite-Rules:** Nach DB-Rollback nochmals `flush_rewrite_rules(true)` für sauberen Zustand.

Rollback-Zeitaufwand: ~3 Minuten (mysqldump-pre ~500 KB).
