## §1 Verständnis

### Ausgangslage
`/praxis/` (page-id 9671, Template `template-standard.php`) zeigt Hero,
Editor-Content und optional zwei CTA-Buttons. **Im Content-Body 0 Links zu
`/leistungen/`** (S20 Services-Hub) **oder `/diagnostik/`** (S21 Diagnostik-Hub).
Die Cluster sind nur über die Top-Nav erreichbar.

### Zielzustand
Auf `/praxis/` erscheinen **nach dem Editor-Content, vor dem optionalen
CTA-Block** zwei Teaser-Cards, die in die Nachbar-Cluster führen:

- **Leistungen** — „Ambulantes Leistungsspektrum — Innere Medizin, Diagnostik,
  Check-ups." → `/leistungen/`
- **Diagnostik** — „Sonographie, EKG, Lungenfunktion, Labor — alles unter einem
  Dach." → `/diagnostik/`

### Constraints
- `template-standard.php` wird auch von `/impressum/`, `/datenschutz*/`,
  Team-narrativ-Pages genutzt — Cross-Links MÜSSEN per Guard auf `/praxis/`
  limitiert sein.
- Keine Werbung, keine Heilversprechen (HWG / CW-005). Texte rein informativ.
- Nutzt vorhandene Component-Library (`.pxz-card`, `.pxz-eyebrow`,
  `.pxz-title-3`, Design-Tokens).

## §2 Lösungsdesign

### Architektur
| Artefakt | Rolle |
|---|---|
| `template-parts/praxis-crosslinks.php` | Template-Part, rendert 2 Cards. Datenliste im Top des Files als PHP-Array (keine externe Datenquelle nötig — 2 Links ändern sich selten). |
| `template-standard.php` | Guard-Aufruf `if ( is_page('praxis') ) { get_template_part( 'template-parts/praxis-crosslinks' ); }` nach dem Content-Block, vor dem CTA-Block. |
| `assets/css/standard.css` | Neuer Block am Ende: `.pxz-praxis-crosslinks*`. 2-Column-Grid, mobile stack. Hover-Lift (translateY(-2px) + shadow). |
| `functions.php` | `PXZ_VERSION 2.7.18 → 2.7.19` (Cache-Buster). |

### Warum `is_page('praxis')` und nicht Custom-Field
- Es gibt genau **eine** Seite, die diese Cross-Links zeigen soll. Custom-Field
  wäre Overkill und birgt Drift-Risiko (Feld gesetzt → plötzlich erscheinen
  Cards auch auf Impressum).
- `is_page()` ist slug-basiert, robust gegen ID-Wechsel.

### Warum Template-Part statt Inline
- Datenliste + Markup in eigenem File = klar auditierbar.
- Wiederverwendbar für spätere Adaptionen (andere Hub-Seiten).
