## §1 Verständnis

### IST (vor N-6.3)

- `cw-transfer push wp:template` funktioniert (N-1, S29): baut aus `trunk/content/team/*.yaml` eine JSON-Datei `inc/data/team.json` und schreibt sie in das Praxis-Theme. CW-008-Backup vor Update. PHP-Side konsumiert die JSON zuerst, Inline-Array als Fallback.
- `cw-transfer diff shopify:page` (N-6, S26) und `diff shopify:template` (N-6.2, S28) liefern read-only Drift-Reports für Shopify.
- `cw-transfer diff wp:template` **fehlt** — `DIFF_TOOLS` enthält nur die beiden Shopify-Einträge.
- Konsequenz: Nach einem Trunk-Edit kann der Operator nicht vor `push wp:template` read-only prüfen, ob der Push tatsächlich eine Änderung bewirken würde.

### Probleme, die N-6.3 löst

1. **Pre-Push-Verifikation für WP-Pattern-B fehlt:** `team-to-wp.mjs` schreibt destruktiv auf das Theme-Filesystem. Ohne Diff kann der Operator keinen No-Op-Push erkennen; jeder Push erzeugt ein CW-008-Backup, auch wenn nichts geändert wird.
2. **Drift-Erkennung für WP-Theme-Daten:** Wenn `inc/data/team.json` im Theme-Repo manuell verändert wird (Verstoß gegen CW-001), muss das read-only erkannt werden können — analog N-6.2 für Shopify.
3. **Backup-Häufigkeit senken:** Ein Diff vor dem Push verhindert No-Op-Writes und damit unnötige Backups.
4. **Symmetrie der Adapter-Suite:** Nach N-6.3 ist der Diff-Quadrant vollständig über beide Plattformen (Shopify + WP) und beide Patterns (A + B reverse). Verbleibende Lücke nur noch `wp:page` (nicht in Scope) und `shopify:product` (N-6.4, eigene Front).
5. **Skalierbarkeit für N-1c:** Spätere Renderer (Services-Hub, Diagnostik-Data) werden denselben Diff-Pfad erben, sobald sie als `wordpress.template.*` in der Registry stehen.

### Nicht-Scope (MVP)

- `wp:page` — WP-Page-Diff ist eine eigene Front (braucht WP-REST-Auth).
- Media-ID-Resolver — image_id bleibt `0` wie IST (N-1b, eigene Front).
- Mehrere Renderer gleichzeitig — N-6.3 liefert exakt einen Renderer (`team`), aber die CLI-Arg-Grammatik ist so gewählt, dass spätere Renderer ohne Breaking Change angehängt werden können.
- Visueller Diff gerenderter Seiten — Review-Pipeline (separat).
- Write-Pfad, Restore-Helper, Auto-Push — außerhalb Diff-Quadrant.

## §2 Lösungsdesign

### F-1 Datenfluss

```
(keine einzelne YAML, sondern Renderer-Handle)
cw-transfer diff wp:template team
  ↓ spawn: bun adapters/wordpress/build-team.mjs
  ↓ stdout (capture) — identisch zum Push-Pfad (N-1)
Local-Payload  { asset: { path: "inc/data/team.json", value: "<JSON-string>\n" },
                 meta:  { source_count, generated_at, renderer } }
                                                                                  |
THEME_PATH env (Default: Local-WP Praxis-Theme auf Cluster-Mini-02)               |
  ↓ resolve(theme_path, asset.path)                                               |
  ↓ readFileSync(<abs>) → live_raw                                                |
Live-File  { exists: true/false, raw: "<string>" | null, size: <bytes> | null }   |
                                                                                  v
                    Parse both JSON strings (beide entstehen aus JSON.stringify)
                                          ↓
                    Canonical-JSON (sorted keys rekursiv) + structural summary
                                          ↓
                    Text-or-JSON diff output
                                          ↓
                    Exit 0 (identical) / 1 (diff or absent) / 2 (error)
```

### F-2 Renderer-Handle-Grammatik (CLI-Arg)

Anders als bei `diff shopify:template <yaml>` nimmt `diff wp:template` **keinen YAML-Pfad** als Argument, weil ein WP-Template-Renderer über *mehrere* YAMLs aggregiert. Stattdessen wird ein **Renderer-Handle** (String) übergeben:

```
cw-transfer diff wp:template <renderer>
```

**Initial registrierte Handles (N-6.3):**

| Handle | Build-Tool | Target-Path | Registry-Eintrag |
|---|---|---|---|
| `team` | `adapters/wordpress/build-team.mjs` | `inc/data/team.json` | `wordpress.template.praxis` (N-1, stable) |

**Dispatch im Adapter:**
```js
const DIFF_RENDERERS = {
  team: { build_tool: "adapters/wordpress/build-team.mjs" }
};
```

**Zukunftssichere Erweiterung (nicht in N-6.3 implementiert, nur Vorbereitung):**
Wenn N-1c kommt (`services`, `diagnostik`), reicht ein neuer Eintrag in `DIFF_RENDERERS` + neues `build-<name>.mjs`. Keine Signatur-Änderung, kein Breaking Change für Operator.

### F-3 Vergleichsregeln

WP-Template-Daten sind (aktuell) ein JSON-Array von Member-Objekten mit stabilen Keys. Die Vergleichs-Logik spiegelt N-6.2, angepasst auf Array-Struktur statt Sections-Map:

| Feld / Signal | Vergleich | Hinweis |
|---|---|---|
| `asset.path` | string-equal | implizit durch Build festgelegt; für Summary relevant |
| `team_json` (ganzes Array) | Canonical-JSON-Vergleich mit sortierten Keys rekursiv | Primäres Drift-Signal |
| `member_count` | number-equal | High-Level, redundant, aber Summary-Relevanz |
| `member_slugs_sorted` (Multiset) | JSON-equal der sortierten Slug-Liste | Zeigt, ob Members hinzugefügt/entfernt wurden |
| `live.size`, `live.mtime` | ignoriert | Filesystem-Metadaten, nicht trunk-relevant |

**Canonical-JSON-Definition:** identisch zu N-6.2 — alle Objekt-Keys werden rekursiv sortiert, Array-Reihenfolge bleibt erhalten.

**Text-Level-Offset:** `firstDiffOffset()` auf den canonical-JSON-Strings — gleicher Helper wie N-6.2, gleiche Semantik.

### F-4 Spezialfall: Live-File existiert nicht

Wenn `<theme>/<asset.path>` nicht existiert (`existsSync` = false): Output `ABSENT — push would CREATE`, Exit-Code 1 (= Diff-vorhanden, Push wäre nicht no-op). Symmetrisch zu N-6.2-ABSENT.

### F-5 Kein Header-Comment-Strip

Im Gegensatz zu Shopify (das beim Asset-Serve einen `/* … */` Header voranstellt) ist das WP-Theme-Filesystem roh — was in die Datei geschrieben wurde, kommt 1:1 zurück. `build-team.mjs` produziert bereits byte-stabil (JSON.stringify + pretty + trailing newline). Es gibt keinen Header zu strippen. Der canonical-JSON-Vergleich kompensiert außerdem reine Whitespace-Differenzen (falls die Datei per Tool reformatiert wurde).

### F-6 Output-Modi

**Default (text):** human-lesbar, kompakt:
```
spec             : wp:template
renderer         : team
source           : trunk/content/team/*.yaml (8 files)
theme_path       : /Users/.../themes/<tenant-theme>
target_path      : inc/data/team.json
live_file        : exists (size 2797 bytes, mtime 2026-04-23T08:12:34Z)
member_count     : EQUAL (local=8, live=8)
member_slugs     : EQUAL (8 slugs: ["dr-aleite","dr-grigorov",…])
team_json        : EQUAL (2796 chars canonical)
RESULT           : EQUAL
```

**JSON-Modus (`FORMAT=json`):** strukturiert für Tooling:
```json
{
  "spec": "wp:template",
  "renderer": "team",
  "source": "trunk/content/team/*.yaml",
  "asset_path": "inc/data/team.json",
  "theme_path": "/Users/.../themes/<tenant-theme>",
  "live": { "exists": true, "size": 2797, "mtime": "..." },
  "fields": {
    "member_count":        { "equal": true, "local": 8, "live": 8 },
    "member_slugs_sorted": { "equal": true, "local": ["dr-aleite","…"], "live": ["dr-aleite","…"] },
    "team_json":           { "equal": true, "local_length": 2796, "live_length": 2796, "first_diff_offset": null }
  },
  "result": "EQUAL",
  "diff_count": 0
}
```

### F-7 Read-Only-Garantie (CW-001/008)

`diff-team.mjs` darf NUR `readFileSync` / `existsSync` / `statSync` aufrufen — keine `writeFileSync`, `mkdirSync`, `unlinkSync`, `copyFileSync`, `renameSync`. Selbsttest in §4 prüft das per Grep. Auch der Build-Sub-Prozess ist read-only (`build-team.mjs` schreibt nur stdout, kein Filesystem-Write).

### F-8 Theme-Path-Resolution

Identisch zu `team-to-wp.mjs`:
1. Wenn `THEME_PATH` env gesetzt und Verzeichnis existiert → nehmen.
2. Sonst: Default-Konstante `DEFAULT_THEME_PATH` (Local-WP Praxis-Theme auf Cluster-Mini-02).

Diese Duplikation zu `team-to-wp.mjs` ist bewusst akzeptiert — eine zentrale `resolveThemePath()`-Helper-Funktion ist ein Folge-Refactor (analog `resolveLiveThemeId` bei N-6.2 offen gelassen).

### F-9 Architektur-Konsequenz

`DIFF_TOOLS` in `tools/cw-transfer` wächst um einen Eintrag:
```js
const DIFF_TOOLS = {
  "shopify:page":     "adapters/shopify/diff-page.mjs",
  "shopify:template": "adapters/shopify/diff-template.mjs",
  "wp:template":      "adapters/wordpress/diff-team.mjs"   // NEU (N-6.3)
};
```

Help-Text wird erweitert: „Diff currently supported for: shopify:page, shopify:template, wp:template".

**Arg-Konvention für `diff wp:template`:** `<renderer-handle>` (nicht YAML-Pfad). `cmdDiff` im CLI bleibt dispatch-generisch — das Argument wird 1:1 an das Tool durchgereicht; `diff-team.mjs` interpretiert es als Renderer-Handle.

### F-10 Kein Wrapper-Shell-Script

Analog N-6 und N-6.2: `cw-transfer diff wp:template team` ist idiomatisch und kurz; ein `tools/diff-team-wp.sh` wäre Boilerplate-Drift.

### F-11 Pattern-Wiederverwendung

N-6.3 wendet das in Session 26 etablierte Pattern **`build-then-fetch-then-diff`** auf eine dritte Dimension an:

| Adapter | „Fetch"-Quelle | „Fetch"-Technik |
|---|---|---|
| N-6 `diff-page.mjs` | Shopify-Admin-API | `GET /pages/<id>.json` |
| N-6.2 `diff-template.mjs` | Shopify-Admin-API | `GET /themes/<id>/assets.json` |
| **N-6.3 `diff-team.mjs`** | **Local-Filesystem** | **`readFileSync`** |

Der Pattern-Eintrag `Nexus/_memory/patterns/build-then-fetch-then-diff.md` wird um einen Abschnitt „Filesystem-Variante" ergänzt.

## §3 Umsetzung

### T-1 Spec
Diese Datei.

### T-2 `adapters/wordpress/diff-team.mjs` (NEU)

- Imports: `node:child_process` (spawnSync), `node:path` (resolve), `node:fs` (existsSync, readFileSync, statSync)
- Argv: `<renderer-handle>` — erlaubt nur registrierte Handles (initial: `team`)
- Schritt A: `spawnSync("bun", [build_tool])` mit `stdio: ["ignore", "pipe", "inherit"]`
  - Bei Sub-Exit ≠ 0: Exit 2 mit Fehlerweiterleitung
  - stdout = Payload-JSON mit `{ asset: { path, value } }`
- Schritt B: `JSON.parse(stdout)` → `localPayload`. Parse `asset.value` als JSON → `localData` (JS-Array für `team`).
- Schritt C: Theme-Path-Resolve (F-8).
- Schritt D: `targetPath = resolve(themePath, asset.path)`. Path-Escape-Guard (analog team-to-wp): muss unter themePath bleiben.
- Schritt E: Wenn `!existsSync(targetPath)` → ABSENT-Pfad (DIFF, Exit 1, kein Lese-Versuch).
- Schritt F: `readFileSync(targetPath, "utf8")` → `liveRaw`. `JSON.parse(liveRaw)` → `liveData`.
- Schritt G: Canonical-JSON von `localData` und `liveData` bilden (sorted keys rekursiv — Arrays bleiben geordnet).
- Schritt H: Field-Compare (F-3). `member_count`, `member_slugs_sorted`, `team_json`.
- Schritt I: Output je nach `process.env.FORMAT` (text default, json optional).
- Exit-Code: 0 wenn alle Felder equal, 1 wenn ≥ 1 Diff oder ABSENT, 2 bei Error.

### T-3 `tools/cw-transfer` (MOD)

- `DIFF_TOOLS["wp:template"] = "adapters/wordpress/diff-team.mjs"`
- `printHelp()` Block aktualisieren: „Diff currently supported for: shopify:page, shopify:template, wp:template"
- Env-Block für `diff wp:template`: `THEME_PATH`, `FORMAT=json`
- Spec-Referenz: `N-6.3_cw-transfer-diff-wp-template.md`
- `cmdDiff` bleibt unverändert (dispatch-generisch, Argument wird 1:1 durchgereicht)

### T-4 Header-Doku in `diff-team.mjs`

Selbst-erklärend: Env-Vars, Exit-Codes, Read-Only-Garantie, Beispiel-Aufruf, Verweis auf Spec. Anti-Pattern-Warnung: kein Write-Call (CW-008).

### T-5 KEIN Wrapper-Shell-Script

Siehe F-10.

### T-6 Member-Slug-Extraktion (Generizität)

Weil der Renderer heute ein Array von Objekten mit `slug`-Key liefert, ist `member_slugs_sorted` trivial. Sobald N-1c Services/Diagnostik kommt, wird der Renderer-Output eventuell ein Objekt mit anderer Struktur haben. N-6.3 implementiert `member_slugs_sorted` **spezifisch für den `team`-Handle** (Array + `slug`-Feld). Bei neuen Handles entscheidet die Spec des jeweiligen Renderers, welches Summary-Feld analog ist. Der `team_json`-Canonical-Compare ist hingegen generisch.

## §6 Pattern-Bezug

N-6.3 wendet das Pattern `build-then-fetch-then-diff` (`Nexus/_memory/patterns/build-then-fetch-then-diff.md`, S26) auf eine dritte Dimension an: Local-Filesystem statt HTTP-API. Der Pattern-Eintrag wird um einen Abschnitt **„Filesystem-Variante"** erweitert (5 Punkte: Fetch = `readFileSync`, ABSENT-Branch = `existsSync`, kein Header-Strip nötig, kein Auth, Unit-of-Write = ganze Datei).

Damit ist das Pattern nicht mehr „zwei Adapter für Shopify", sondern **„allgemeines Build-then-Fetch-then-Diff-Muster, angewandt auf drei Quellen"**. Das ist die stabile, skalierbare Basis für künftige N-6.x-Adapter (z. B. `wp:option`, `ios:bundle`).

## §7 Abgrenzung zu N-1 / N-6.2 (Schwester-Adapter)

| Adapter | Richtung | Schreibt? | Scope |
|---|---|:---:|---|
| N-1 `build-team.mjs` | Trunk → Stdout | ❌ read-only build | Payload-Renderer für N-1-Push und N-6.3-Diff |
| N-1 `team-to-wp.mjs` | Stdin → Theme-FS | ✅ mit CW-008-Backup | Destruktiver Write auf `inc/data/team.json` |
| N-6.2 `diff-template.mjs` | Trunk ↔ Shopify-Theme-Asset | ❌ read-only | Pattern B (Shopify-seitig) |
| **N-6.3 `diff-team.mjs`** | **Trunk ↔ WP-Theme-File** | **❌ read-only** | **Pattern B reverse (WP-seitig) — Gegenstück zu N-1-Push** |

N-6.3 ist das diff-Pendant zu `team-to-wp.mjs` (push) — Diff-Quadrant der WP-Pattern-B-Adapter ist damit symmetrisch. `wp:page` (Pattern A, braucht WP-REST) und `shopify:product` (N-6.4) bleiben die letzten offenen Diff-Quadranten.
