## §1 Verständnis

### IST (vor N-6.2)

- `cw-transfer diff shopify:page <yaml>` funktioniert (N-6, S26). Exit 0 EQUAL, 1 DIFF, 2 ERROR.
- `cw-transfer push shopify:template <yaml>` funktioniert (S22). Schreibt Trunk → Theme-Asset (Pattern B).
- `cw-transfer pull shopify:template <name>` funktioniert (S22). Liest Theme-Asset → Proto-Trunk.
- `cw-transfer diff shopify:template <yaml>` ist **nicht registriert** — Dispatch-Table `DIFF_TOOLS` enthält nur `shopify:page`.
- Konsequenz: Für Pattern-B-Pages (Template-basiert, z. B. Juvantis `/pages/uber-uns`) kann der Operator den Trunk-Stand NICHT read-only gegen Shopify prüfen, bevor er `push shopify:template` auslöst.

### Probleme, die N-6.2 löst

1. **Pre-Push-Verifikation für Pattern-B fehlt:** Bei Template-basierten Pages macht der Adapter `template-to-shopify.mjs` einen destruktiven PUT auf `/themes/<id>/assets.json`. Ohne Diff kann der Operator nicht vorher wissen, ob der Push überhaupt etwas ändert.
2. **Drift-Erkennung für Theme-Assets:** Wenn im Theme-Editor (oder per Shopify-CLI) ein Template manuell verändert wird (Verstoß gegen CW-001), muss das read-only erkannt werden können.
3. **Backup-Häufigkeit senken:** `template-to-shopify.mjs` legt bei jedem Update ein Backup an (CW-008). Mit `diff` entfällt der No-Op-Push und damit das unnötige Backup.
4. **Symmetrie der Adapter-Suite:** Nach N-6 und N-6.2 sind `push`/`pull`/`diff` vollständig symmetrisch über beide Shopify-Content-Patterns (A=Page, B=Template). Quadrant geschlossen.

### Nicht-Scope (MVP)

- `shopify:product` — Produkt-Diff hat eigene Felder (Variants, Images, Metafields). Folge-Front (N-6.4).
- `wp:page` — symmetrische Diff-Logik für WP-Seiten. Folge-Front (N-6.3, setzt N-1 voraus).
- Visueller Diff von gerenderten Seiten (Puppeteer-Screenshots) — Review-Pipeline (separat).
- Multi-Template-Diff in einem Aufruf — ein YAML erzeugt genau ein Template-Asset. Mehrere YAMLs werden per Shell-Loop sequentiell behandelt.

## §2 Lösungsdesign

### F-1 Datenfluss

```
Trunk-YAML (z. B. trunk/content/pages/_shared/ueber-uns.yaml)
  ↓ spawn: bun adapters/shopify/build-template.mjs <yaml>
  ↓ stdout (capture)
Local-Payload  { asset: { key: "templates/page.uber-uns.json", value: "<JSON-string>" } }
                                                                                  |
SHOPIFY_STORE + SHOPIFY_ADMIN_TOKEN (+ SHOPIFY_THEME_ID optional)                 |
  ↓ GET /admin/api/<v>/themes.json  (only when no SHOPIFY_THEME_ID) → role=main   |
  ↓ GET /admin/api/<v>/themes/<id>/assets.json?asset[key]=<key>                   |
Live-Asset  { key, value: "/* header */\n<JSON-string>", size, updated_at, ... }  |
                                                                                  v
                    Parse both JSON (with header-comment strip on live side)
                                          ↓
                    Canonical-JSON (sorted keys) + structural summary
                                          ↓
                    Text-or-JSON diff output
                                          ↓
                    Exit 0 (identical) / 1 (diff or absent) / 2 (error)
```

### F-2 Vergleichsregeln

Templates sind JSON-Objekte mit variabler Struktur, aber stabilem Schema:

```json
{
  "sections": {
    "<section-id>": { "type": "<section-type>", "settings": { ... }, "blocks": { "<block-id>": { "type": "...", "settings": { ... } } }, "block_order": [ ... ] }
  },
  "order": [ "<section-id>", ... ]
}
```

| Feld / Signal | Vergleich | Hinweis |
|---|---|---|
| `asset.key` | string-equal | implizit identisch, weil aus derselben Page-ID abgeleitet — aber Selbstcheck lohnt |
| `sections` (Map) | canonical-JSON-Vergleich mit sortierten Keys | Map-Reihenfolge darf differieren ohne als Diff zu zählen |
| `order` (Array) | strikt positions-gleich | Reihenfolge ist semantisch (Sections werden so gerendert) |
| `section_count` | number-equal | High-Level-Signal, redundant zu `sections`-Map, aber für Summary hilfreich |
| `section_types` (sortiert, Multiset) | JSON-equal der sortierten Liste | zeigt ob Sections ausgetauscht wurden |
| `asset.updated_at` / `size` / `public_url` | ignoriert | site-bestimmt, nicht trunk-relevant |

**Canonical-JSON-Definition:** `JSON.stringify(obj, Object.keys(obj).sort())` rekursiv — alle Objekt-Keys sortiert. Array-Reihenfolgen bleiben unverändert. So wird ein struktureller Diff (gleiche Inhalte, andere Key-Reihenfolge im Shopify-Roundtrip) NICHT als Diff gezählt.

**Text-Level-Offset:** Auf den canonical-JSON-Strings der beiden Seiten wird `firstDiffOffset()` (wie in N-6) angewandt, damit der Operator einen einzelnen Char-Index bekommt, an dem die Differenz beginnt. Primär für Debugging.

### F-3 Spezialfall: Live-Asset existiert nicht

Wenn `GET /themes/<id>/assets.json?asset[key]=<key>` 404 wirft: Output „ABSENT — push would CREATE", Exit-Code 1 (= Diff-vorhanden, weil ein Push nicht no-op wäre). Analog N-6.

### F-4 Header-Comment-Stripping (Shopify-Auto-Annotation)

Shopify prependet beim Lesen von Template-JSON einen auto-generierten Comment:
```
/* This template was created by the Theme Editor ... */
{ "sections": ... }
```
`extract-template.mjs` strippt diesen bereits. `diff-template.mjs` MUSS denselben Strip auf der Live-Seite anwenden, SONST ist jeder Diff trivialerweise positiv. Identisches Regex wie in `extract-template.mjs`: `^\/\*[\s\S]*?\*\/\s*/m`.

Die lokale Seite (aus `build-template.mjs`) enthält diesen Comment nicht — `render*`-Funktionen im Lib-Code produzieren reinen JSON-String.

### F-5 Output-Modi

**Default (text):** human-lesbar, kompakt:
```
cw-transfer diff shopify:template trunk/content/pages/_shared/ueber-uns.yaml
> spec             : shopify:template
> source           : trunk/content/pages/_shared/ueber-uns.yaml
> asset_key        : templates/page.uber-uns.json
> live_theme_id    : 150231367947
> live_asset       : exists (size 8421 bytes, updated 2026-04-22T20:41:13Z)
> section_count    : EQUAL (3)
> section_types    : EQUAL (["main-page","text-block","team-grid"])
> order            : EQUAL (3 items)
> template_json    : DIFFER (local 8390 chars, live 8421 chars, first diff at offset 2134)
RESULT           : DIFF (1 field)
```

**JSON-Modus (`FORMAT=json`):** strukturiert für Tooling-Integration:
```json
{
  "spec": "shopify:template",
  "source": "trunk/content/pages/_shared/ueber-uns.yaml",
  "asset_key": "templates/page.uber-uns.json",
  "live": { "exists": true, "theme_id": 150231367947, "size": 8421, "updated_at": "2026-04-22T20:41:13Z" },
  "fields": {
    "section_count": { "equal": true, "local": 3, "live": 3 },
    "section_types": { "equal": true, "local": ["main-page","text-block","team-grid"], "live": ["main-page","text-block","team-grid"] },
    "order": { "equal": true, "local_length": 3, "live_length": 3 },
    "template_json": { "equal": false, "local_length": 8390, "live_length": 8421, "first_diff_offset": 2134 }
  },
  "result": "DIFF",
  "diff_count": 1
}
```

### F-6 Read-Only-Garantie (CW-001/008)

`diff-template.mjs` darf NUR `client.get()` aufrufen — keine `post`/`put`/`delete`. Selbsttest in §4 prüft das per `grep`. Auch der Build-Sub-Prozess ist read-only (`build-template.mjs` schreibt nur stdout, keine API-Aufrufe).

### F-7 Theme-ID-Resolution

Identisch zu `template-to-shopify.mjs` und `extract-template.mjs`:
1. Wenn `SHOPIFY_THEME_ID` gesetzt und numerisch → nehmen.
2. Sonst: `GET /themes.json` + filter `role === "main"` → live-Theme.
3. Wenn kein Theme mit `role=main`: Exit 2.

Diese Dreifach-Duplikation ist bewusst akzeptiert — eine zentrale `resolveLiveThemeId()`-Helper-Funktion ist ein Folge-Refactor (nicht N-6.2-Scope).

### F-8 Architektur-Konsequenz

`DIFF_TOOLS` in `cw-transfer` wächst um einen Eintrag:
```js
const DIFF_TOOLS = {
  "shopify:page":     "adapters/shopify/diff-page.mjs",
  "shopify:template": "adapters/shopify/diff-template.mjs"
};
```
Help-Text wird erweitert: „Diff currently supported for: shopify:page, shopify:template".

### F-9 Kein Wrapper-Shell-Script

Analog N-6: `cw-transfer diff shopify:template <yaml>` ist idiomatisch und kurz; ein `tools/diff-template-shopify.sh` wäre Boilerplate-Drift.

## §3 Umsetzung

### T-1 Spec
Diese Datei.

### T-2 `adapters/shopify/diff-template.mjs` (NEU)

- Imports: `node:child_process` (spawnSync), `node:path` (resolve), `./lib/shopify-rest-client.mjs`
- Argv: `<trunk-yaml-path>`
- Schritt A: `spawnSync("bun", ["adapters/shopify/build-template.mjs", yamlPath])` mit `stdio: ["ignore", "pipe", "inherit"]`
  - Bei Sub-Exit ≠ 0: Exit 2 mit Fehlerweiterleitung
  - stdout = Payload-JSON mit `{ asset: { key, value } }`
- Schritt B: `JSON.parse(stdout)` → `local.asset.{key, value}`. Parse `value` als JSON → `localTemplate`.
- Schritt C: Theme-ID-Resolve (F-7).
- Schritt D: `GET /themes/<id>/assets.json?asset[key]=<url-encoded-key>`. Bei 404 → ABSENT-Pfad (DIFF, Exit 1).
- Schritt E: Live-value: Header-Comment strippen (F-4), dann JSON-Parse → `liveTemplate`.
- Schritt F: Canonical-JSON von `localTemplate` und `liveTemplate` bilden (sorted keys rekursiv).
- Schritt G: Field-Compare (F-2). `section_count`, `section_types` (sortiert), `order` (positional), `template_json` (canonical-String + first_diff_offset).
- Schritt H: Output je nach `process.env.FORMAT` (text default, json optional).
- Exit-Code: 0 wenn alle Felder equal, 1 wenn ≥ 1 Diff oder ABSENT, 2 bei Error.
